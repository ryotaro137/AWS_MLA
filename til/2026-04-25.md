### ECRとECSの基本コンポーネントとユースケース

#### Amazon ECR (Elastic Container Registry) のコンポーネント
Amazon ECR は、開発者が Docker ボックスイメージを簡単に保存、管理、デプロイできるフルマネージド型のコンテナイメージレジストリサービスです。
- **レジストリ**: AWS アカウントがアクセスできる Amazon ECR レジストリはそれぞれ 1 つですが、レジストリ内に複数のイメージリポジトリを作成してイメージを保存できます。
- **認可トークン**: Docker クライアントはイメージをプッシュおよびプルする前に、Amazon ECR レジストリに AWS ユーザーとして認証される必要があります。`aws ecr get-login-password` コマンドにより、Docker に渡すための認証情報を入手できます。
- **リポジトリ**: Amazon ECR イメージリポジトリには Docker イメージが含まれています。
- **リポジトリポリシー**: リポジトリポリシーを使用して、リポジトリとその中にあるイメージへのアクセスを制御できます。
- **イメージ**: リポジトリに Docker イメージをプッシュおよびプルすることができます。開発システムでこれらのイメージをローカルに使用したり、Amazon ECS タスク定義で使用したりできます。

**ECRの具体的なユースケース**:
* **CI/CDパイプラインとの統合**: AWS CodePipeline や GitHub Actions などの CI/CD ツールから、ビルドされたコンテナイメージを自動的に ECR にプッシュし、一元管理する。
* **機械学習モデルのデプロイ**: Amazon SageMaker の学習用・推論用カスタムコンテナイメージを保存し、各タスク実行時にプルして利用する。
* **安全なイメージ共有**: リポジトリポリシーや IAM を活用し、組織内の特定チームや別 AWS アカウントに対してのみ特定のコンテナイメージへのアクセスを許可する。

#### Amazon ECS (Elastic Container Service) のコンポーネント
Amazon ECS は、コンテナ化されたアプリケーションを簡単にデプロイ、管理、スケーリングできる可用性に優れたコンテナオーケストレーションサービスです。
- **コンテナインスタンス**: Amazon ECS エージェントを実行し、クラスターに登録されている Amazon EC2 インスタンス。
- **コンテナ**: タスクの一環として作成された Linux コンテナ。
- **クラスター**: タスクを配置できるコンテナインスタンスの論理的なグループ。
- **タスク定義、タスク**: 1 つ以上のコンテナの定義を含むアプリケーションの説明。タスク定義を単一のインスタンスにしたものを「タスク」と呼びます。
- **サービス**: Amazon ECS サービスでは、指定された数のタスク定義のインスタンスを同時に実行および維持できます。

#### Amazon ECS タスクのスケジューリングとサービス管理
Amazon ECS は、共有状態のオプティミスティック同時実行システムであり、タスクやコンテナについて柔軟なスケジューリングを可能にします。Amazon ECS スケジューラでは、Amazon ECS API により提供されるクラスターの状態に関する情報を活用して、適切な配置を決定します。
- **サービススケジューラー**: 長時間実行されているステートレスサービスとアプリケーションでサービススケジューラを実行できます。指定した数のタスクを継続的に実行でき、タスクが失敗したとき（例えば、基盤となるコンテナインスタンスが何らかの理由で失敗した場合）には再スケジュールできます。
- **ロードバランシング**: オプションとして、タスクを Elastic Load Balancing (ELB) に対して登録し、トラフィック管理を行うように設定できます。
- **サービスの更新**: 新しいタスク定義のデプロイ、希望するタスクの実行数の変更など、サービススケジューラによって管理されるサービスを更新できます。
- **サービス定義パラメータ**: どのタスク定義をサービスで使用するか、そのタスクのインスタンスをいくつ実行するか、どのロードバランサーを使用するかを定義します。

**ECSの具体的なユースケース**:
* **マイクロサービスアーキテクチャの運用**: web サーバー、API アプリケーション、バックグラウンドワーカーなどを個別のコンテナ（タスク）として ECS クラスター上で実行し、ALB のパスベースルーティングと組み合わせてサービスを展開する。
* **バッチ処理の設定**: 定期実行が必要なデータ処理タスクや機械学習のバッチ推論ジョブを、EventBridge と組み合わせて ECS タスクとしてスケジュール実行する（AWS Fargate との組み合わせが一般的）。
* **ブルー/グリーンデプロイ**: AWS CodeDeploy と連携し、ECS サービス上で稼働するアプリケーションのアップデート時に、ダウンタイムなしに安全な切り替え（ロールバック含む）を実現する。

### ECRとECSの違いについて

一言で言うと以下のようになります：
* **ECR (Elastic Container Registry)** は、コンテナの「**保管庫（倉庫）**」です。
* **ECS (Elastic Container Service)** は、コンテナの「**実行・管理環境（エンジン）**」です。

#### 役割の違いを比較

| 項目 | Amazon ECR (保管庫) | Amazon ECS (実行・管理) |
| :--- | :--- | :--- |
| **主な役割** | コンテナイメージを安全に保存・管理する | ECRなどからイメージを取り出し、実際に実行・稼働させる |
| **現実世界に例えると** | アプリの「設計図」や「インストールディスク」をしまっておく倉庫 | 倉庫から設計図を取り出して、実際の「工場」として稼働させる仕組み |
| **主に行う操作** | `docker push`（保存）、`docker pull`（取り出し） | コンテナを何個動かすかの設定、停止した際への再起動設定など |
| **AWS内の連携** | CodePipelineからのビルド結果を受け取る、ECSやSageMakerにイメージを提供する | ALB（ロードバランサー）と連携してトラフィックをさばく、Fargate上で動かす |
| **料金の発生ポイント** | 保存しているイメージの**データ容量**と転送量 | 起動している**コンテナの実行時間（CPU/メモリ量）**やEC2インスタンス料金 |

#### 一連の流れでの関係性

一般的なデプロイ作業は次のような流れになります。

1. **手元のPC（またはCI/CD環境）**でアプリを作り、Dockerイメージを作成（ビルド）する。
2. 作ったイメージを **Amazon ECR** に「**保存（Push）**」する。
3. **Amazon ECS** に対して、「ECRにある、このイメージを使ってコンテナを立ち上げて」と「**指示**」を出す。
4. **ECS** は **ECR** からイメージをダウンロード（Pull）し、実際にアプリケーションを「**実行**」して稼働させ続ける。

#### MLA-C01 / 実務でのポイント
* **SageMakerとの連携**: 機械学習の実務やMLA試験において、カスタマイズした学習アルゴリズムや推論コードのDockerイメージは **ECRに保存** します。その後、SageMakerが学習ジョブや推論エンドポイントを立ち上げる際に、**ECRからイメージをPullして実行**するという流れになります（この場合、実行環境はECSではなくSageMakerになります）。
