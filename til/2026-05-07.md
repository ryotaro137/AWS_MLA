### FSx for Lustre / S3 Transfer Acceleration / SageMaker 高速ファイルモード の違い
- **概要**: 名前に「高速」「S3」が含まれ混同しやすいが、目的のフェーズが全く異なる3つの仕組み。
- **試験ポイント**: 「コスト効率よく学習データを読み込むには？」という設問でFSxと高速ファイルモードの選択が問われやすい。FSxは最高性能だが常時課金、高速ファイルモードは追加コストなしで学習開始を早められる。
- **3つの役割**:
  - **FSx for Lustre**: 高性能並列ファイルシステムをSageMaker学習インスタンスにマウント。S3と同期してランダムアクセス・大量並列I/Oに対応。VPC接続必須・ファイルシステム分の常時課金が発生。
  - **S3 Transfer Acceleration**: CloudFrontエッジ経由でS3へのアップロード/ダウンロードを高速化。学習とは無関係で「データをS3に置くフェーズ」の話。地理的に遠い場所からの転送で効果大。
  - **SageMaker 高速ファイルモード（Fast File Mode）**: 全データをダウンロードせずにS3からストリーミングで学習開始できるSageMaker専用の入力モード。追加インフラ不要。逐次読み取り時に最も効果的。
- **使い分け**:
  - S3へのデータ転送が遅い → **S3 Transfer Acceleration**
  - 学習中のI/Oがボトルネック（ランダムアクセス多い） → **FSx for Lustre**
  - コスト重視で学習開始を早めたい・逐次読み取り中心 → **SageMaker 高速ファイルモード**
- **実務での使い方**: 大規模MLトレーニングではFSx for Lustreが鉄板。コスト最適化が求められる場合は高速ファイルモードを検討。S3 Transfer Accelerationはデータ収集・入稿フェーズのインフラ設計で使う。
- **副業での需要**: ★★★ MLOpsのインフラ設計・コスト最適化案件で差別化できる知識。

### AWS Glue DataBrew：プロファイルジョブ vs レシピジョブ
- **概要**: DataBrewの2種類のジョブ。目的が「評価」か「変換」かで明確に異なる。
- **試験ポイント**: プロファイルジョブはデータを**変更しない**（読み取り専用）。「検出・評価」→プロファイル、「実行・変換」→レシピ、の使い分けが問われやすい。
- **2つの役割**:
  - **プロファイルジョブ**: データの統計的プロファイリングを実行。欠損値の割合・値の分布・外れ値・カラム間の相関・データ型の不一致などを統計レポート（JSON）として出力。データ自体は変更しない。「何を変換すべきか判断するためのインプット」。
  - **レシピジョブ**: プロファイル結果をもとに定義した変換手順（レシピ）を実際のデータに適用し、変換済みデータセットをS3等に出力。欠損値補完・正規化・フィルタリングなどを実行。
- **使い分け**:
  - データ品質の問題を**検出**したい → **プロファイルジョブ**
  - 欠損値補完・正規化を**実行**したい → **レシピジョブ**
- **実務での使い方**: まずプロファイルジョブでデータ品質を把握し、問題箇所を特定してからレシピを作成→レシピジョブで変換というワークフローが標準。
- **副業での需要**: ★★★ データエンジニアリング・ML前処理自動化案件で活用できる知識。

### SageMaker Model Monitor vs SageMaker Clarify
- **概要**: どちらもモデルの品質に関わるサービスだが、監視タイミングと目的が異なる。Model Monitorは「本番稼働中のドリフト検出」、Clarifyは「バイアス検出と説明可能性」に特化。
- **試験ポイント**: Model Monitorは**学習過程ではなく本番エンドポイント**を監視する点が問われやすい。データ品質全般（欠損値・外れ値）はDataBrew/Data Wranglerの担当で、Clarifyではない。
- **2つの役割**:
  - **Model Monitor**: 本番デプロイ後のエンドポイントを継続監視。推論データをベースライン（学習時データ）と比較してドリフトを検出・アラート。監視種別は①データ品質（統計的ドリフト）②モデル品質（精度劣化）③バイアスドリフト④特徴量帰属ドリフト（SHAP値）の4種類。
  - **Clarify**: バイアス検出とモデルの説明可能性に特化。学習前（訓練データのバイアス）・学習後（モデル予測のバイアス）・本番（Model Monitorと連携したバイアスドリフト）の3タイミングで機能する。
- **データ品質評価の担当サービス整理**:
  - 欠損値・外れ値・分布の評価 → **DataBrew プロファイルジョブ** / **Data Wrangler**
  - 本番データのドリフト検出 → **Model Monitor**
  - バイアス・説明可能性 → **Clarify**
- **実務での使い方**: MLOpsパイプラインではModel MonitorをエンドポイントにアタッチしてCloudWatchアラームと連携、品質劣化を自動検知して再学習トリガーを組む構成が標準。
- **副業での需要**: ★★★★ MLOps・モデル運用監視の設計案件で必須知識。Model Monitorの設定ができるエンジニアは希少で差別化になる。
