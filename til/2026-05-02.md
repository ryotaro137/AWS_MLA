## 学習内容

### Amazon Forecast による時系列予測
- **概要**: MLを使った完全マネージドの時系列予測サービス。ARIMA・DeepAR+・Prophet・NPTS・CNN-QR の組み込みアルゴリズムを提供。前処理・特徴量エンジニアリング・学習・評価・デプロイを自動化し、最小限の開発工数で需要予測・在庫管理・リソース計画などを実装できる。**Holidays featurization** 機能で祝日の影響を自動組み込み可能
- **試験ポイント**: 複数の関連する時系列データを同時に扱い、相関・季節性・トレンドを自動処理したい場合は **DeepAR+** が正解。Holidays featurization は DeepAR+ と組み合わせることで祝日効果も自動考慮できる。SageMaker の DeepAR（単体）との違いに注意——Forecast の DeepAR+ はマネージドでデプロイまで一貫して自動化される
- **実務での使い方**: 小売の需要予測・EC の在庫補充計画・エネルギー消費量予測など、複数SKUや複数拠点の時系列を一括して予測するユースケースで採用。AutoML的に複数アルゴリズムを並列評価してベストモデルを選ぶ「AutoPredictor」機能も活用される
- **副業での需要**: ★★★ 小売・物流・エネルギー系クライアントの需要予測システム構築案件で引き合いあり。コード不要で高精度予測を実現できるため、既存SaaS予測ツールからの移行案件でも採用が増加中

### SageMaker Studio ドメインと部門別ルーティング設計
- **概要**: SageMaker Studio ドメインはユーザー・アプリ・リソースを管理する論理単位。部門・チームごとに独立したVPC設定・IAMロール・セキュリティ設定を持てる。**CreatePresignedDomainUrl API** を使うと指定ドメインへの有効期限付き署名付きURLを生成でき、プロキシアプリから部門ごとの適切なドメインへユーザーをルーティングできる
- **試験ポイント**: 混同しやすいAPI3種の使い分けが頻出。**CreatePresignedDomainUrl**（Studio ドメイン用）・**CreatePresignedNotebookInstanceUrl**（旧来のnotebookインスタンス用、Studioドメインには不可）・**CreateHumanTaskUi / ListHumanTaskUis**（Ground Truth ラベリングUI用、ルーティングには使えない）。部門別ルーティングが要件なら必ず CreatePresignedDomainUrl を選ぶ
- **実務での使い方**: IdP（Active Directory・Okta・Azure AD）でSSO認証 → プロキシアプリがユーザーの部門情報を取得 → DynamoDB から部門ID↔ドメインID のマッピングを参照 → CreatePresignedDomainUrl API で署名付きURLを生成してリダイレクト、という動的ルーティングパターンが企業向けML基盤の標準構成
- **副業での需要**: ★★★ 複数部門が共有するML基盤の設計・構築案件でドメイン分離設計の知識が必須。SSO統合とセキュアなアクセス制御を組み合わせた要件は大企業向け案件で頻出

### VPCエンドポイントによるプライベートネットワーク構成
- **概要**: VPCエンドポイントはインターネットを経由せずVPC内リソースからAWSサービスへアクセスする機能。**ゲートウェイエンドポイント**（S3・DynamoDBのみ、ルートテーブルにルート追加）と**インターフェースエンドポイント**（その他AWSサービス全般、VPC内にENIを作成しプライベートIPでアクセス）の2種類がある
- **試験ポイント**: 「インターネットを経由せずプライベートにAWSサービスへアクセスしたい」要件では **NATゲートウェイは不正解**（NATはインターネット経由のため）。正解はVPCエンドポイント。S3へはゲートウェイエンドポイント、SageMaker等その他サービスへはインターフェースエンドポイント。なおS3はVPC内に配置できないリージョンレベルのサービスである点も注意
- **実務での使い方**: SageMakerノートブックインスタンスをプライベートサブネットに配置するセキュア構成では、S3用ゲートウェイエンドポイント＋SageMaker用インターフェースエンドポイントをセットで設定するのが標準。すべてのトラフィックをAWSネットワーク内に閉じてコンプライアンス要件に対応する
- **副業での需要**: ★★★★ セキュリティ要件の厳しいML基盤（医療・金融・行政）でVPCエンドポイント設計は必須スキル。ネットワーク設計ができるMLエンジニアは希少で単価差別化につながる

### Kinesis Data Streams + Firehose + Lambda によるリアルタイムストリーミング処理
- **概要**: **Kinesis Data Streams** は高スループット・低レイテンシのストリーミングデータ取り込みサービス（シャード単位でスケール、最大7日間保持、複数コンシューマー対応）。**Kinesis Data Firehose** はストリームデータを自動でS3・Redshift・Elasticsearch等に配信するフルマネージドサービスで、バッファリング・圧縮・変換を自動処理。**Lambda** と統合することでレコード単位のリアルタイム変換も可能
- **試験ポイント**: リアルタイムストリーミングデータの変換→S3保存が要件の場合は **Kinesis Data Streams → Firehose（Lambda変換付き）→ S3** が正解パターン。**Glue ジョブは不正解**——バッチ処理向けで起動・実行に時間がかかりリアルタイム要件を満たせず、Firehoseからの直接読み取りも不可。「運用オーバーヘッド最小」という条件もFirehoseを支持する根拠になる
- **実務での使い方**: IoTセンサーデータや推論ログをリアルタイムで収集・変換してS3データレイクに蓄積し、定期的にMLモデルの再学習に利用するMLOpsパイプラインで定番の構成。Firehoseの最大5分バッファリングとLambda変換を組み合わせてフィールド正規化・フィルタリングを行う
- **副業での需要**: ★★★★ リアルタイムデータ基盤構築案件でKinesisは最頻出サービスの一つ。ML推論結果のリアルタイム収集・モニタリング基盤の設計でも必須知識
- **補足（Firehose直接S3接続）**: Firehose は Kinesis Data Streams を経由せず**直接S3に書き込める**。Data Streams を中間に挟むとレイテンシとコストが増加するだけなので、単純な「収集→S3保存」ならFirehose単体で完結させるのがシンプルで効率的。逆に Data Streams 単体は S3 に直接書き込めない点も注意

### Amazon API Gateway の役割と Lambda との組み合わせ
- **概要**: API Gateway は HTTP/HTTPS リクエストの受付・ルーティング・認証・レート制限を担う「玄関口」、Lambda は実際のビジネスロジックを実行する「処理室」。2つが補完し合うことでサーバーレスなAPIバックエンドが完成する。両者ともトラフィックに応じて自動スケールし、使った分だけ課金のためアイドルコストがゼロ
- **試験ポイント**: OSI参照モデルでは API Gateway は主に **L7（アプリケーション層）** に相当。ただし厳密にはL5〜L7にまたがる——WebSocket での接続管理（L5）・TLS終端とJSONマッピング（L6）・HTTPメソッド/パス/ヘッダーの解釈とルーティング（L7）。ALBとの違いは「APIのビジネスロジック寄りの制御（認証・変換・レート制限）ができる」点
- **実務での使い方**: 生成AI対話UIのバックエンドでは REST API（同期）または WebSocket API（双方向リアルタイム）を用途に応じて選択。Lambda と組み合わせてサーバー管理ゼロで本番APIを構築するパターンが中小規模のML推論エンドポイントや社内ツールで標準的
- **副業での需要**: ★★★★ サーバーレスAPI構築はほぼすべてのWeb系・ML系案件で求められる基本スキル。API Gateway + Lambda の設計・実装ができると幅広い案件に対応できる

### SageMaker Feature Store：GetRecord vs BatchGetRecord
- **概要**: SageMaker Feature Store のオンラインストアから特徴量を取得するAPIが2種類ある。**GetRecord**：レコード識別子1件を指定して単一レコードを取得、常に最新バージョンを返す、低レイテンシでリアルタイム推論向け。**BatchGetRecord**：複数のレコード識別子をまとめて1回のAPI呼び出しで取得、バッチ推論や複数レコード一括取得に効率的
- **試験ポイント**: リアルタイム推論で「最新の特徴量を1件・低レイテンシで取得」したい場合は **GetRecord が正解**。BatchGetRecord は複数レコードを返すため最新レコードを特定するための追加フィルタリングが必要になり、単一レコード取得には非効率。「最小限の処理で最新レコードを取得」という条件がGetRecord を支持する根拠
- **実務での使い方**: オンライン推論エンドポイント（Lambda + SageMaker Endpoint）でリクエストごとにユーザー特徴量をリアルタイム取得する際に GetRecord を使用。バッチ変換ジョブや複数ユーザーの特徴量を一括取得する前処理では BatchGetRecord で効率化する
- **副業での需要**: ★★★ MLOps・特徴量管理基盤の構築案件で Feature Store の知識は差別化ポイント。リアルタイム推論基盤設計では API の使い分けまで問われることが多い

### PaddleOCR vs マネージドOCRサービスの使い分け
- **概要**: PaddleOCR はオープンソースのOCRライブラリ。カスタム要件への柔軟な対応が可能だが、SageMaker 上で使用する場合はモデルの訓練・デプロイ・メンテナンスをすべて自前で行う必要があり運用負荷が高い
- **試験ポイント**: 「低メンテナンス・運用オーバーヘッド最小」が要件の場合、PaddleOCR は不正解。OCR ユースケースではマネージドサービスの **Amazon Textract**（文書からテキスト・表・フォームを自動抽出）が正解になる。「カスタムモデルが必要 → PaddleOCR等」「すぐ使えて運用不要 → Textract」という切り分けが重要
- **実務での使い方**: 一般的な文書・帳票OCRは Textract で十分。特殊なフォント・独自レイアウト・日本語手書きなど Textract が対応しきれない要件の場合のみ PaddleOCR 等のカスタムモデルを採用する
- **副業での需要**: ★★★ 文書デジタル化・帳票自動処理案件で OCR 技術の選定知識が必要。マネージドサービス優先でコストと運用負荷を抑える提案ができると評価が高い

### VPCインターフェイスエンドポイントのセキュリティ制御3層構造
- **概要**: VPCインターフェイスエンドポイント（AWS PrivateLink）はVPC内のサブネットにENIを作成し、インターネット・NAT・VPN不要でAWSサービスへプライベート接続する。SageMaker Service API もインターフェイスエンドポイントに対応済み。アクセス制御は **IAMポリシー・VPCエンドポイントポリシー・セキュリティグループ** の3層で構成される
- **試験ポイント**: 3つの制御レイヤーの役割と違いが頻出。**IAMポリシー**（誰が何をできるか）・**VPCエンドポイントポリシー**（エンドポイント経由のアクセスをリソースベースで制御、デフォルト全許可）は**AND条件**——両方が許可して初めてアクセス可能。**セキュリティグループ**（ENIへのL4トラフィック制御、許可ルールのみ）はENIに直接適用。**NACL**はサブネットレベル・ステートレス・拒否ルールあり——ENIに直接適用できないためエンドポイントの細粒度制御には不向き
- **実務での使い方**: SageMaker をプライベートVPC内で使用する構成では、エンドポイントポリシーで「特定のIAMロールからのみ許可」＋セキュリティグループで「SageMaker Studio のサブネットからのみインバウンド許可」という2段階制御が標準的なセキュアアーキテクチャ
- **副業での需要**: ★★★★ セキュリティ・コンプライアンス要件の厳しいML基盤設計案件でPrivateLink＋多層アクセス制御の知識は必須。IAMだけでなくネットワーク層まで設計できるエンジニアは希少で高単価につながる
