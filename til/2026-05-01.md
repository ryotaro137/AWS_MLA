## 学習内容

### IAMポリシー条件（Condition）によるアクセス制御
- **概要**: IAMポリシーに条件式を追加し、特定の条件を満たす場合のみアクセスを許可する仕組み。代表的な条件キーとして `aws:sourceIp`（送信元IPアドレス）、`aws:sourceVpc`（送信元VPC）、`aws:PrincipalTag`（IAMプリンシパルのタグ）などがある
- **試験ポイント**: VPN経由アクセスの制御では `aws:sourceVpc` は不適切。VPNクライアントはVPC外部から接続するため、VPCベースの検証が機能しない。正解は `aws:sourceIp` でVPNクライアントのIPアドレスを制限すること。署名付きURLの悪用防止にも有効
- **実務での使い方**: SageMaker Studio等の署名付きURL発行を伴うサービスへのアクセスを社内VPN経由に限定する際、`aws:sourceIp` 条件でVPN出口のIPレンジをホワイトリスト化するパターンが一般的
- **副業での需要**: ★★★ セキュリティ強化・コンプライアンス対応案件でIAMポリシー設計は必須スキル。「最小権限の原則」を条件キーで実装する要件がML基盤構築案件でも頻出

### Amazon API Gateway WebSocket API + AWS Lambda によるリアルタイム対話基盤
- **概要**: API Gateway WebSocket API はクライアント・サーバー間のリアルタイム双方向通信をマネージドで提供するサービス。Lambda と組み合わせることで、サーバーレスかつスケーラブルなバックエンドを構築でき、チャットや対話型AIインターフェースに最適
- **試験ポイント**: リアルタイム双方向通信が要件の場合は REST API ではなく WebSocket API を選択。Lambda は接続管理・メッセージルーティング・スケーリングをすべてマネージドに扱えるため、運用オーバーヘッド最小という要件でも正解になりやすい
- **実務での使い方**: 生成AIチャットボットや対話型MLデモの構築で API Gateway WebSocket + Lambda のパターンが定番。接続ごとに `connectionId` を管理し、DynamoDB に保存して非同期でメッセージをプッシュする構成が多い
- **副業での需要**: ★★★★ 生成AI系チャットアプリ・対話型インターフェース案件で需要急増中。サーバーレス構成で運用コストを抑えたい中小企業向け開発でもよく採用される

### Amazon SageMaker Ground Truth によるラベリング基盤
- **概要**: 高品質なラベル付きデータセットを作成するマネージドサービス。画像分類・物体検出・セグメンテーション・テキスト分類など多様なタスクに対応。自動ラベリング（Active Learning）と人間レビューを組み合わせてコスト効率よく高精度なアノテーションを実現する
- **試験ポイント**: 機密データのラベリングには **プライベートワーカーフォース** を選択する。パブリックワーカー（Amazon Mechanical Turk）はデータが外部に渡るため不適切。プライベートワーカーフォースでは社内従業員のみアクセス可能なワーカーチームを構成し、データ機密性を担保できる点が頻出
- **実務での使い方**: 医療・金融など機密性の高いドメインのMLプロジェクトで社内アノテーター向けにプライベートワーカーフォースを設定するパターンが多い。レビュープロセス（複数ワーカーによる多数決・監査）と組み合わせてラベル品質を管理する
- **副業での需要**: ★★★ データアノテーション基盤の設計・構築案件で需要あり。特に業界特化型MLプロジェクト（医療画像・製造業の品質検査など）でGround Truthを活用したパイプライン構築の引き合いが増加中

### AWS Glue Relationalize変換 ＋ PySpark によるネストJSON処理
- **概要**: Glue の `Relationalize` 変換は、ネストされたJSON（配列・オブジェクト）を自動でフラットなリレーショナルテーブルに分解するマネージド機能。親子関係を保ちながら複数テーブルに分割し、MLアルゴリズムが扱える表形式に変換できる。GlueのPySparkジョブはサーバーレス・自動スケーリングで、カスタムETLスクリプトをS3↔DB間で実行できる
- **試験ポイント**: ネストJSONを最小限のカスタムコードで表形式へ変換する要件では **Glue Relationalize が正解**。EMR Serverless も Spark 実行基盤だが、変換ロジックを一から自前実装する必要があり開発・運用コストが高い。「マネージド標準機能で運用オーバーヘッドを最小化」という文脈では Glue を選ぶ
- **実務での使い方**: API ログや IoT データなど深くネストされた JSON を ML 特徴量エンジニアリング用にフラット化する前処理パイプラインで活用。Glue Crawler でスキーマ検出 → Relationalize 変換 → S3（Parquet）出力 → SageMaker 学習というパターンが一般的
- **副業での需要**: ★★★★ データ基盤構築・ETLパイプライン設計案件で Glue は最頻出サービスの一つ。特に非構造化JSONを扱う案件では Relationalize の知識が差別化ポイントになる

### AWS KMS ＋ Encryption CLI によるファイル暗号化とアクセス制御
- **概要**: AWS KMS はHSMで保護された暗号化キーをマネージドで管理するサービス。S3・EBS・RDS等のAWSサービスと統合済み。KMSキーへのアクセス権限は **IAMポリシー＋KMSキーポリシー** の2層で制御できる。AWS Encryption CLI はKMSキーを使いコマンドラインからファイルを暗号化・復号するツール
- **試験ポイント**: 「ファイルを暗号化し、どのアプリが鍵にアクセスできるか制御したい」という要件の正解は **「KMSキーを作成 → Encryption CLIで暗号化」**。引っかかりやすい誤答の識別法：SSHキー（サーバー認証用）・API Gatewayキー（APIアクセス制御用）・IAMロール（権限付与のみ、暗号化操作は不可）はいずれもデータ暗号化には使えない
- **実務での使い方**: MLパイプラインで機密トレーニングデータをS3に保存する際、KMSカスタマーマネージドキー（CMK）を作成し、キーポリシーでSageMaker実行ロールのみアクセス許可するパターンが標準。Encryption CLIはローカルファイルをS3アップロード前に暗号化するスクリプトに組み込まれることが多い
- **副業での需要**: ★★★ セキュリティ・コンプライアンス要件のあるML基盤案件で必須知識。医療・金融分野では特に暗号化とアクセス制御の設計が求められる

### AWS Glue DataBrew によるノーコードデータ準備
- **概要**: コードなしで視覚的UIからデータクリーニング・正規化・変換を実行できるマネージドサービス。S3・Redshift・RDS・JDBC接続など多様なデータソースに対応。**データセット**（入力元定義）→ **レシピ**（変換手順の定義・再利用可能）→ **レシピジョブ**（変換実行・S3出力）の3要素で構成される
- **試験ポイント**: ジョブの種類の使い分けが頻出。**プロファイルジョブ**はデータ品質の統計分析・可視化のみで変換は不可。**レシピジョブ**が実際のクリーニング・変換を実行する。また S3 は JDBC 接続不可（オブジェクトストレージのためJDBC非対応）→ S3からデータセットを作成する場合はS3パスを直接指定する
- **実務での使い方**: MLパイプラインの前処理フェーズで、データエンジニアリングの経験が浅いチームメンバーがデータ準備を担当する際に活用。レシピのバージョン管理により変換処理の再現性・監査証跡も確保できる
- **副業での需要**: ★★★ データ品質改善・前処理自動化案件で需要あり。ノーコードツールのため非エンジニア向け社内データ基盤構築案件でも採用が増加中
