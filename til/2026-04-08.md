## 学習内容
### Amazon Mechanical Turk とプライベートワークフォース
- **概要**: 不特定多数のパブリックワーカーにタスク（ラベリング等）を依頼できるクラウドソーシングプラットフォーム。
- **試験ポイント**: 機密データを含む「プライベートデータセット」の要件がある場合、Mechanical Turk（パブリックワーカー）は不適切. セキュリティリスクを回避するためには、従業員等で構成される「プライベートワークフォース」の使用が正解となる。
- **実務での使い方**: SageMaker Ground Truth等でラベリングを行う際、データの機密性に応じて「パブリック（Turk）」「ベンダー」「プライベート（自社）」のワークフォースを使い分ける。機密データはインターネット越しに不特定多数に公開してはならない。
- **副業での需要**: ★☆☆☆☆ (直接的な案件は少ないが、Ground Truthを用いたアノテーション工程の設計において、セキュリティ基盤の提案として重要)

### Amazon Comprehend Medical
- **概要**: 医療テキスト（処方箋、臨床ノート等）から保護された健康情報（PHI）を自動的に検出し、マスキングできる医療特化型の自然言語処理サービス。
- **試験ポイント**: HIPAA等のコンプライアンス要件を満たす設計。通常のComprehendとは異なり、医療ドメイン特有のエンティティ（薬剤、診断名、検査結果等）を高い精度で抽出できる点が重要。
- **実務での使い方**: 臨床ノートや医療記録の匿名化（レッドアクション）を自動化したり、非構造化データから医療インサイトを抽出してBIツールや他サービスと連携させるデータパイプラインに組み込む。
- **副業での需要**: ★★☆☆☆ (医療系の高度な専門性が求められるが、セキュリティとコンプライアンスが重視されるヘルスケア領域のデータ基盤構築で差別化ポイントとなる)

### SageMaker Model Monitor と Pipelines でのベースライン管理
- **概要**: デプロイ済みモデルの品質（データ、モデル、バイアス、特徴量属性）を継続的に監視するサービス。SageMaker Pipelinesと連携し、再学習時のベースライン自動更新も可能。
- **試験ポイント**: ベースラインは `statistics.json`（統計）と `constraints.json`（制約）で構成される。ドリフト検出時の違反内容は `constraint_violations.json` に記録される。Pipelinesでは `QualityCheckStep` や `ClarifyCheckStep` を使用し、`register_new_baseline=True` 設定で再学習時のベースライン更新を制御する。
- **実務での使い方**: ドリフト検知をトリガーに再学習パイプラインを自動実行する仕組みを構築する。再学習の際、新しい学習データの特性に合わせて `register_new_baseline=True` でベースラインを更新することで、古い制約による誤検知を防ぐ。
- **副業での需要**: ★★★★☆ (MLの問題解決だけでなく「運用し続ける」ためのMLOps基盤構築として、商用レベルの案件で非常に高く評価されるスキル)

### AWS Glue Relationalize と EMR Serverless の比較
- **概要**: ネストされたJSONデータをリレーショナルな表形式にフラット化する変換手法。Glueの `Relationalize` 変換はこれを標準機能として提供する。
- **試験ポイント**: 「ネストされたJSONのフラット化」かつ「実装・運用工数の最小化」という要件に対する最適解は Glue `Relationalize`。EMR Serverless はサーバーレス Spark 実行環境だが、フラット化ロジックの自前実装と保守（開発オーバーヘッド）が必要になる点が比較で問われやすい。
- **実務での使い方**: 複雑な階層構造を持つJSONデータをS3から読み込み、Glue PySparkジョブ内で `Relationalize` を適用。親子関係を維持したまま複数のテーブルに自動分割・フラット化し、機械学習アルゴリズムが扱える形式でS3へ出力するパイプラインを最小工数で構築する。
- **副業での需要**: ★★★☆☆ (データ変換フェーズの効率化は、開発期間の短縮と品質安定に直結するため、データ基盤構築案件での有力な提案材料になる)

### ストリーミングデータ処理（Flink, MSK, Kinesis）
- **概要**: リアルタイムで生成されるデータを収集・処理・分析するためのフルマネージドサービス群。
- **試験ポイント**: 「ストリーミングデータに対してSQLを実行したい」「Jupyterノートブック等で対話的に分析したい」という要件には **Managed Service for Apache Flink** が最適。Kinesis Data Firehoseは配信・変換（Lambda経由）は得意だが、SQLクエリの直接実行やインタラクティブ分析機能を持たない点が比較のキモ。
- **実務での使い方**: Kinesis Data Streams や MSK で高スループットなデータを取り込み、Flink を使って時間窓（Windowing）集計や状態管理を行い、リアルタイムダッシュボードで可視化する。複雑なイベント時間処理（Event Time Processing）が必要なケースで採用する。
- **副業での需要**: ★★★★☆ (リアルタイムログ分析や異常検知基盤の構築は、ビジネス価値に直結しやすく専門性も高いため、高単価かつ差別化しやすい案件領域)

### SageMaker Model Registry と推論エンドポイントへの展開
- **概要**: モデルのバージョン管理、承認フローを統合管理するサービス（Model Registry）と、承認済みモデルをリアルタイム推論用にホスティングする機能（Endpoints）。
- **試験ポイント**: 「再学習後のモデルを安全にデプロイしたい」という要件では、Model Registry での「承認（Approved）」プロセスが必須工程として問われる。承認ステータスをトリガーに CD パイプラインを動かし、エンドポイントを自動更新（Blue/Green デプロイ等）する流れが重要。
- **実務での使い方**: SageMaker Pipelines の最終ステップでモデルを Registry に登録。メトリクスを確認した管理者が承認することで、EventBridge 等を介して実際のデプロイ処理を実行させる。これにより、未検証モデルの誤デプロイを防止し、トレーサビリティを確保する。
- **副業での需要**: ★★★★★ (商用 ML システムにおけるモデルガバナンスとデプロイ自動化は必須要件であり、この一連の CI/CD 基盤を構築できるスキルは極めて評価が高い)

### 対話型インターフェース構築のサービス選定
- **概要**: リアルタイム双方向通信や会話型UIを実現するためのAWSサービス。用途に応じた適切な選定（WebSocket, Lex, QuickSight等）が重要。
- **試験ポイント**: 「生成AIを用いたオープンエンドな対話UI」や「双方向通信」の要件に対する正解選定。QuickSight（BI/可視化）やLex（定型対話用）は自由な生成AIチャットには不向き。WebSocket API + Lambda の構成が、低オーバーヘッドかつ柔軟な対話基盤として問われやすい。
- **実務での使い方**: API Gateway WebSocket API を入口とし、Lambda 経由で Amazon Bedrock 等の生成AIと連携。WebSocket の双方向性を活かして、AI の回答をストリーミング形式で即座にユーザーへ表示することで、ユーザー体験（UX）を向上させる実装が主流。
- **副業での需要**: ★★★★★ (生成AIチャットボット開発は極めて需要が高く、API Gateway と Lambda によるサーバーレスかつリアルタイムな実装スキルは、高単価な開発案件で必須となる)

### SageMaker ネイティブフレームワークと BYOC の比較
- **概要**: 学習・推論環境のパッケージング手法。主要な ML フレームワーク（PyTorch, TensorFlow, XGBoost等）は AWS が事前構築済みコンテナを提供している。
- **試験ポイント**: 「最小の運用オーバーヘッドで既存スクリプトを移行したい」という要件では、BYOC ではなく **ネイティブフレームワーク（スクリプトモード）** が正解。BYOC は柔軟性は高いが、Amazon ECR への管理やパッチ適用等の「ヘビーリフティング」が自社責任（高オーバーヘッド）になる点が比較で頻出。
- **実務での使い方**: 基本は事前構築済みイメージを使用し、`requirements.txt` で追加ライブラリを管理する。どうしてもOSレベルの特殊な依存関係や独自バイナリが必要な場合のみ BYOC を選択し、保守コストを許容する設計判断を下す。
- **副業での需要**: ★★★★☆ (既存資産のクラウド移行案件において「迅速かつ運用負荷が低い」構成を提案できる能力は、顧客の満足度と保守性に直結する重要スキル)

### SageMaker マルチモデル(MME) vs マルチコンテナエンドポイント
- **概要**: 1つのエンドポイントで複数のモデルをホストし、リソース効率とコストを最適化する手法。
- **試験ポイント**: 「同一フレームワークの大量（数千〜）のモデル」を低コストでホストしたい場合は **MME** が最適。一方、「異なるフレームワークの混在」や「推論パイプライン（前処理＋推論等）」を構成したい場合は **マルチコンテナ** が正解。MME の動的なロードによる「コールドスタート」の可能性が比較で問われやすい。
- **実務での使い方**: ユーザーごとに細かいモデルを出し分ける SaaS 等では MME を使い、インフラコストを最小化する。前処理（Scikit-learn）から推論（PyTorch）へデータを渡す一連のフローを一本化したい場合はマルチコンテナで「推論パイプライン」を構築し、API 呼び出しを簡素化する。
- **副業での需要**: ★★★★☆ (インフラコストの削減は実務案件で非常に喜ばれる提案。MME によるリソース集約の提案は、顧客のランニングコスト削減に直結するため価値が高い)

### 欠損値補完（Imputation）の基本
- **概要**: カテゴリ変数や数値変数に含まれる欠損値を、一定の統計的ルールに基づいて推定値で埋める前処理手法。
- **試験ポイント**: データの型に応じた適切な手法の選定が重要。平均値（Mean）は一般的な数値、中央値（Median）は外れ値が強い数値データ、最頻値（Mode）はカテゴリデータの補完に適している点が頻出。
- **実務での使い方**: SageMaker Data Wrangler 等でデータ分布を確認し、偏りが大きい（外れ値がある）場合は中央値を採用するなど、モデルのバイアスを抑える前処理を設計する。機械学習のデータ品質担保における基本工程。
- **副業での需要**: ★★★☆☆ (データクレンジングは ML 開発の基礎であり、適切な補完手法を根拠を持って選択・実装できるスキルは、実務レベルのデータエンジニアとして必須)

### 特徴量エンジニアリング（外れ値・対数変換・エンコーディング・正規化）
- **概要**: 機械学習モデルの学習効率と精度を向上させるためのデータ変換技術。外れ値処理、歪度修正、カテゴリ数値化等を含む。
- **試験ポイント**: 「右へ偏った分布を正規分布に近づける対数変換」「外れ値を除去・クリッピングして回帰直線の歪みを防ぐ手法」が頻出。Min-Max 正規化は外れ値により他のデータが狭い範囲に圧縮される弱点がある。
- **実務での使い方**: 住宅面積など大きな値が分布を歪める場合に、対数変換を適用してスケールを均一化する。カテゴリ変数はワンホットエンコーディングで数値化し、数値データには適用しない。
- **副業での需要**: ★★★★☆ (適切な特徴量設計ができるエンジニアは、モデル精度向上への貢献度が大きく、高単価なデータサイエンス・分析案件で重宝される)

### 時系列予測（DeepAR）とデータ変換（Glue/Parquet）
- **概要**: 大規模な関連時系列データの学習に特化したアルゴリズム（DeepAR）と、入力を最適化する列指向フォーマット（Parquet）および ETL サービス（Glue）。
- **試験ポイント**: 「DeepAR の入力形式（JSON Lines, Parquet）と圧縮（Gzip, Snappy）」の選択が重要。Parquet が列指向で大規模処理に効率的な理由や、Glue ジョブで出力形式や圧縮コーデックをパラメータ指定できる点などが頻出。
- **実務での使い方**: S3 上の膨大な JSON ログを Glue で Parquet（Snappy 圧縮）に効率化した後、DeepAR を用いて複数商品の需要予測モデルを構築する。`train`（必須）と `test`（オプション）のデータチャネルを適切に設定し、学習を回す。
- **副業での需要**: ★★★★☆ (時系列データ分析や需要予測は AI/ML 分野で不動のニーズがあり、Glue を活用した前処理から DeepAR による学習までを一貫して設計できるスキルは付加価値が高い)

### レコメンデーションアルゴリズム（Factorization Machines vs k-NN）
- **概要**: ユーザーとアイテムの相関を学習し、推薦やランキングを行うためのアルゴリズム選定。
- **試験ポイント**: 「高次元のスパース（疎）なデータ」かつ「レコメンデーション」が要件にある場合は **Factorization Machines (FM)** が正解。一方、k-NN は類似ポイントの検索には向くが、ユーザー・アイテム間の複雑な相互作用をモデル化するレコメンデーションには適さない点が比較で頻出。
- **実務での使い方**: 購買履歴やクリックログなど、大半の要素が 0 であるスパースな入力データから、FM を用いてユーザーに最適なアイテムを推薦するシステムを構築する。レコメンデーション、CTR 予測、ランキング等のタスクで第一選択肢となる。
- **副業での需要**: ★★★★☆ (レコメンデーション機能の実装は EC やメディア系サービスの案件で非常にニーズが強い。スパースデータの特性を理解したアルゴリズム選定ができることは、実務レベルの ML エンジニアとしての付加価値になる)

### AWS Glue によるデータ品質管理とメタデータ自動収集
- **概要**: 機械学習用のデータソースにおいて、信頼性を担保する品質監視機能（Data Quality）と、スキーマ管理を自動化するツール（Crawler）。
- **試験ポイント**: 「DQDL を用いた品質ルール定義」や「クローラーによるスキーマ・パーティション推論」が頻出。Data Quality がサーバーレスで DeeQu を基盤とし、ML による異常検知や推奨ルール生成機能を備えている点が重要。
- **実務での使い方**: クローラーで S3 上の生データからカタログを自動生成し、Data Quality ルールを適用して、欠損値やデータ型の不整合を学習前に自動検知する「守り」のデータパイプラインを構築する。
- **副業での需要**: ★★★★☆ (データの信頼性（Data Observability）は実務で重視される領域。Glue のマネージド機能を活用して低コストかつ高品質な基盤を提案できる能力は高く評価される)

### SageMaker Feature Store と効率的なデータ取り込み
- **概要**: 特徴量を管理・共有するためのマネージド基盤。リアルタイム推論用の「オンラインストア（DynamoDB等）」と学習用の「オフラインストア（S3）」で構成される。
- **試験ポイント**: 「大量の履歴データを効率的に取り込みたい」という要件では、個別の `PutRecord` API ではなく **Spark コネクタ**（EMR 等）による並列処理が最適。オンライン/オフライン両方を有効化することで、同一データを低レイテンシ推論とバッチ学習の両方に提供できる点が重要。
- **実務での使い方**: ストリーミングデータは Lambda 等から `PutRecord` でオンラインへ即時反映し、過去分は EMR/Spark でオフラインへ一括ロード。特徴量のバージョン管理や検索機能を活用し、チーム間での特徴量の再利用性を高め、トレーニング/サービング・スキューを防止する。
- **副業での需要**: ★★★★☆ (大規模 ML システムにおいて、特徴量の管理（Feature Management）は開発効率と品質を左右する。Feature Store とビッグデータ基盤（Spark/EMR）を連携させた設計スキルは、高単価なデータ基盤案件で重宝される)

### AWS でのデータ暗号化と適切なキー選定 (KMS vs others)
- **概要**: データの機密性を担保するための暗号化手法と、混同しやすい認証・認可キー（SSH、API Key等）の整理。
- **試験ポイント**: 「ファイルをコマンドラインから暗号化したい」という要件に対し、SSH キーや API キーではなく **AWS Encryption CLI + KMS キー** を正しく選択できるかが頻出。IAM ロールは権限付与には使うが暗号化操作自体は行わない点、KMS キーポリシーで細かいアクセス制御が可能な点も重要。
- **実務での使い方**: 機密性の高いデータセットや ML モデルの重みを扱う際、KMS カスタマーマネージドキーを用いて暗号化し、S3 等に保存。復号権限を特定の IAM ロールにのみ絞り込むことで、万が一のデータ流出時も中身を保護できるセキュアな構成を構築する。
- **副業での需要**: ★★★★☆ (セキュリティ設計は実務での信頼に直結する。単に機能を実装するだけでなく、暗号化とアクセス制御（IAM/KMS）を統合して設計できる能力は、小〜中規模案件でも強く求められる)

### AWS Glue DataBrew による視覚的なデータ準備
- **概要**: コード不要（No-code）でデータのクリーニングや変換ができる GUI 基盤。非技術者でも高度なデータ準備が可能。
- **試験ポイント**: 「プログラムを書かずに重複削除や正規化を行いたい」という要件に対し、Glue ETL ではなく **Glue DataBrew** が最適解。「現状分析のみ」を行う **プロファイルジョブ** と、「実際の変換を実行」する **レシピジョブ** の使い分けが重要。S3 データセット作成時に JDBC は不要（直接パス指定）である点も頻出。
- **実務での使い方**: プロファイルジョブでデータの欠損率や相関を可視化し、課題を特定。GUI で構築した「レシピ（変換手順）」をレシピジョブとしてスケジューリング実行し、常に最新のクレンジング済みデータを S3 へ出力するパイプラインをノーコードで構築する。
- **副業での需要**: ★★★★☆ (データ準備は ML 工程の 8 割を占める。専門家でなくても迅速かつ正確にデータ加工を進められる DataBrew の活用は、スピード重視のプロジェクトで極めて価値が高い)

### SageMaker Studio Domain と動的ルーティング (Presigned Domain URL)
- **概要**: ユーザーやアプリケーションを管理する論理単位。部門ごとにVPCやIAM、リソースを完全に分離して管理可能。
- **試験ポイント**: IdP（認証）、DynamoDB（部門/ドメイン紐付け）、`CreatePresignedDomainUrl`（アクセスURL生成）を組み合わせた「動的ルーティング」の構成。独自のポータルサイト経由でも、適切なドメインへ一時的なセキュアURLで誘導する仕組みが問われる。
- **実務での使い方**: 企業内プラットフォームで複数のプロジェクトや部門が同一AWSアカウント内に混在している場合、ユーザー属性に基づいてアクセス先ドメインを自動的に振り分けるポータル（プロキシアプリ）のバックエンドとして実装する。
- **副業での需要**: ★★★☆☆ (エンタープライズ向けのML開発基盤構築において、ガバナンス（分離）とUX（SSOリダイレクト）を両立させる高度な設計スキルとして評価される)

### SageMaker Studio 関連 API の識別 (Domain vs Notebook vs Ground Truth)
- **概要**: SageMaker Studio（ドメイン）、ノートブックインスタンス、Ground Truth（ヒューマンタスクUI）へのアクセスAPIの役割。
- **試験ポイント**: 「Studio ドメインへのアクセス」には `CreatePresignedDomainUrl` が唯一の正解。`CreatePresignedNotebookInstanceUrl` はノートブックインスタンス用、`CreateHumanTaskUi`/`ListHumanTaskUis` はラベリングタスクのカスタムUI用であり、Studio へのルーティングには使用不可。
- **実務での使い方**: 要件に応じた正確な API 選定を行う。旧来のノートブックインスタンス管理から SageMaker Studio (Evolved) への移行が進む中、マネージドな統合環境へのセキュアなアクセス経路を設計するために必須の知識。
- **副業での需要**: ★★☆☆☆ (正確なサービス・API選定は、実装の手戻りを防ぎセキュリティ要件を満たすための基礎。上位のアーキテクチャ設計案件で必須知識となる)

### SageMaker Clarify と Data Wrangler によるバイアス検出
- **概要**: モデルのバイアス検出と説明性を提供するマネージドサービス（Clarify）と、データ準備を GUI で行うサービス（Data Wrangler）。
- **試験ポイント**: 「データ準備段階（Data Wrangler 内）」での統計的バイアス検出に Clarify が使用される点。ノーコードで視覚的なレポートが得られることが重要。Clarify は学習中やデプロイ後（Model Monitor 連携）でも使用可能。
- **実務での使い方**: 学習データセットの公平性を担保するため、Data Wrangler の分析機能から Clarify を呼び出し、カテゴリごとの不均衡やバイアスを可視化。問題があれば Data Wrangler の変換機能で補正する。
- **副業での需要**: ★★★★☆ (AI 倫理や責任ある AI への関心が高まる中、バイアス検出や説明性レポートの提供は ML 案件での必須要件になりつつある)

### バイアス検出・性能管理サービスの使い分け (Clarify vs Model Monitor vs Ground Truth)
- **概要**: 「いつ」「何を」分析するかに応じたサービスの適切な選定。
- **試験ポイント**: データ準備・学習段階のバイアスは **Clarify/Data Wrangler**。デプロイ後のドリフト（データ/バイアス）監視は **Model Monitor**。**Ground Truth** (ラベリング) や **Experiments** (追跡) にはバイアス検出機能がない点が出題のキモ。
- **実務での使い方**: 開発フェーズでは Clarify でモデルの振る舞いを説明（SHAP値 等）し、実運用フェーズでは Model Monitor でパフォーマンス低下やバイアス再発を監視する多層防御的な運用を設計する。
- **副業での需要**: ★★★☆☆ (適切なサービス選定による MLOps ライフサイクルの設計は、長期運用の信頼性を高める提案として高く評価される)
