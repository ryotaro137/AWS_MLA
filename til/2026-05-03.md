## 学習内容

### ゲートウェイ VPC エンドポイント（Gateway VPC Endpoint）
- **概要**: VPC から Amazon S3・DynamoDB へインターネットを経由せずに安全に接続できる機能。インターネットゲートウェイや NAT デバイスが不要になる。
- **試験ポイント**: ゲートウェイエンドポイントは**同一リージョン内の接続のみ**サポート。S3 データが別リージョンに存在する場合は使用不可。クロスリージョン接続には NAT Gateway＋インターネット経由などが必要。
- **実務での使い方**: プライベートサブネット内の ML ワークロードが S3 の学習データにアクセスする際、NAT コストを削減しつつセキュアに接続するパターンで利用する。ただしクロスリージョン構成では使えないため設計時に要注意。
- **副業での需要**: ★★★ セキュリティ要件の厳しいデータ基盤構築案件で VPC 設計の一部として登場することが多い。

### SageMaker Clarify（バイアス検出・特徴量貢献度）
- **概要**: トレーニングデータ・モデルのバイアス検出と、SHAP値による特徴量貢献度の分析・モニタリングを行うサービス。
- **試験ポイント**: トレーニングデータに**不均衡（クラスの偏り）**がある場合、Clarify の処理ジョブでバイアスを特定するのが正解の選択肢になりやすい。特徴量貢献度のドリフトモニタリングは、推論後にどの特徴量がモデル結果に影響しているかを継続的に監視する用途で使う。
- **実務での使い方**: モデルの公平性監査や説明可能性レポートの作成に利用。本番モデルの予測品質低下の原因調査として、特徴量貢献度のドリフトを定期モニタリングする構成が多い。
- **副業での需要**: ★★★ 金融・医療など規制業種の MLOps 案件で説明可能 AI 対応が求められる際に登場することが多い。

### SageMaker ノートブックと VPC エンドポイントのネットワーク設定
- **概要**: SageMaker ノートブックから VPC エンドポイント経由でサービスにアクセスするには、ノートブック自体を**同一 VPC 内のサブネット**に配置する必要がある。
- **試験ポイント**: パブリックインターネット経由では VPC エンドポイントにアクセスできない。エラーの原因がサブネット・セキュリティグループ・ルートテーブルの設定ミスではなく「ノートブックが VPC 外にある」ことに起因するケースを見抜けるかが問われる。ノートブックインスタンス作成時に VPC・サブネットを指定して内部ネットワーク接続を有効化することが正解。
- **実務での使い方**: プライベート環境で SageMaker を運用する際は、ノートブック・Training Job・Endpoint すべてを同一 VPC 内に統一し、VPC エンドポイント経由でのみ通信させるネットワーク設計が基本。
- **副業での需要**: ★★★ セキュアな ML 基盤構築案件で VPC 設計と合わせて必須知識となる。

### SageMaker Pipelines ステップタイプ一覧

#### データ処理・学習系

| ステップ | 対応API | 用途 |
|---------|--------|------|
| `ProcessingStep` | `DescribeProcessingJob` | データ前処理・後処理・モデル評価スクリプトの実行 |
| `TrainingStep` | `DescribeTrainingJob` | モデルの学習（estimator の `.fit()` をパイプライン内で実行） |
| `TuningStep` | `DescribeHyperParameterTuningJob` | ハイパーパラメータチューニング（複数訓練ジョブを並列実行） |
| `AutoMLStep` | `DescribeAutoMLJob` | Autopilot による自動モデル構築 |
| `TransformStep` | `DescribeTransformJob` | 訓練済みモデルによるバッチ推論 |

#### モデル登録・デプロイ系

| ステップ | 対応API | 用途 |
|---------|--------|------|
| `CreateModelStep` | `DescribeModel` | SageMaker Model オブジェクトの作成（Transform・Endpoint デプロイ前に使う） |
| `RegisterModel` | `DescribeModelPackage` | Model Registry へのモデル登録・バージョン管理・承認フロー連携 |

#### 品質・バイアスチェック系

| ステップ | 用途 |
|---------|------|
| `QualityCheckStep` | データ品質・モデル品質のベースライン計算とドリフト検出 |
| `ClarifyCheckStep` | Clarify によるバイアス検出・特徴量貢献度ドリフトの自動検出 |

#### フロー制御系

| ステップ | 用途 |
|---------|------|
| `ConditionStep` | 評価結果に基づくパイプラインの分岐制御（`ConditionGreaterThan` 等を使用） |
| `FailStep` | 条件不合格時にパイプラインを意図的に失敗状態で終了させる |

#### 外部連携系

| ステップ | 用途 |
|---------|------|
| `LambdaStep` | AWS Lambda 関数の呼び出し（軽量処理・外部通知） |
| `EMRStep` | Amazon EMR の Spark ジョブ実行（大規模分散処理） |
| `CallbackStep` | SQS キュー経由で外部処理の完了を待機（SageMaker 外部システムとの統合） |

- **試験ポイント**:
  - `ConditionStep` + `RegisterModel` の組み合わせ（精度閾値を超えた場合のみ登録）が頻出パターン
  - `ClarifyCheckStep`（バイアス・説明可能性）と `QualityCheckStep`（データ・モデル品質）の違いを区別できるようにする
  - `CallbackStep` は SageMaker 外部のシステムと統合する唯一の手段
  - ステップ間のデータ受け渡しは `.properties.ProcessingOutputConfig.Outputs[...].S3Output.S3Uri` などのプロパティ参照で行う
- **実務での使い方**: ProcessingStep → TrainingStep → ConditionStep → RegisterModel の流れが MLOps パイプラインの基本構成。QualityCheckStep・ClarifyCheckStep を挟むことで継続的なモデル品質監視も実現できる。
- **副業での需要**: ★★★★ MLOps 基盤構築案件でパイプライン設計は中核スキル。SageMaker Pipelines の実装経験は単価アップに直結する。
