### Amazon FSx for Lustre

マネージド型の高パフォーマンスファイルシステムであり、高いスループットでのデータ読み書きが可能です。特にハイパフォーマンスコンピューティング（HPC）と機械学習ワークロードに最適化されています。

- **試験での出題ポイント**
  - **S3とのネイティブ連携**: Amazon S3から直接データを読み込めるため、事前にトレーニングデータをコピーする時間を省き、トレーニングの開始時間を大幅に短縮できる点が頻出です。
  - **大規模トレーニングの最適化**: 複数のチューニングオプションが用意されており、大規模なモデルトレーニングにおけるファイルI/Oのボトルネックを解消する用途で出題されます。
- **実務でのユースケース**
  - Amazon SageMakerでの分散トレーニングで、テラバイトやペタバイト級の巨大なデータセット（画像、動画、テキストコーパスなど）を各学習ノードへ高速かつ同時に供給する高速一時ファイルストレージとして利用されます。
- **副業案件での需要度と単価感**
  - 大規模MLモデルの学習環境構築やMLOpsパイプライン設計（単価: 時給4,000円〜8,000円程度）において、パフォーマンス最適化のための知識として役立ちます。小規模案件ではS3直接参照（Fast File Mode等）で済むことが多く、登場頻度はやや限定的です。

### Amazon Aurora と Amazon Redshift のゼロ ETL 統合

完全マネージド型のリレーショナルデータベースである Aurora は、Amazon Redshift とゼロ ETL 統合が可能です。これにより、複雑なデータパイプラインを構築・運用することなく、データを連携できます。

- **試験での出題ポイント**
  - **ほぼリアルタイムの分析と機械学習**: Aurora に書き込まれたデータがほぼリアルタイムで Redshift で利用可能になります。そのため、複雑な ETL 操作を必要とせず、迅速なデータ分析や機械学習の入力データとしての活用が可能になる点がシステム統合の文脈で問われます。
  - **アーキテクチャの運用・開発負担の最小化**: データをストリーミングするための個別データパイプライン（AWS GlueやKinesis等）の構築・維持が不要になるため、開発作業や運用負担を最小限に抑える「最適なアーキテクチャ」として出題される傾向があります。
- **実務でのユースケース**
  - アプリケーション側のトランザクションデータベース（Aurora）で発生する購買データなどを、機械学習の分析基盤や BI ツールから即座に参照し、リアルタイムなレコメンデーションエンジンや需要予測ダッシュボードなどに活用するモダンなデータ基盤として利用されます。
- **副業案件での需要度と単価感**
  - リアルタイム分析基盤の設計・構築案件（単価: 時給4,000円〜7,000円程度）で注目されています。運用のオーバーヘッドを抑えつつリアルタイム性を高めたいクライアントからの引き合いが強く、知っていると提案の幅が広がる技術です。

### Amazon Transcribe Medical

Amazon Transcribe の医療特化型機能であり、医療関連の音声を高精度でテキストに文字起こしできます。さらに、音声ファイル内の PHI（保護すべき医療情報: Protected Health Information）を自動的に識別・墨塗り（マスキング）する機能を備えています。

- **試験での出題ポイント**
  - **PHI の識別と抽出**: 医療音声データ（診察の録音など）から機密性の高い PHI を識別・保護する要件において、複雑なカスタム NLP 処理を作らず、Transcribe Medical の標準機能（一括テキスト変換ジョブにおける PHI 識別機能など）を利用するのが最適解となるシナリオが問われます。
- **実務でのユースケース**
  - 電子カルテ（EHR）システムへの自動入力補助プラットフォームや、遠隔医療アプリでの医師と患者の会話の自動書き起こしなど、個人情報保護規制（HIPAA等）に準拠したセキュアな医療情報のデータ化パイプラインの構築で利用されます。
- **副業案件での需要度と単価感**
  - ヘルスケア業界に特化した技術ですが、厳格な個人情報保護要件が求められるプロジェクト（単価: 時給5,000円〜8,000円など）において、Amazon Comprehend Medical と組み合わせて高度なセキュリティアーキテクチャの提案ができると非常に重宝されます。

### Amazon S3 を基盤とするデータレイクアーキテクチャ

Amazon S3 は、構造化データおよび非構造化データの両方を格納・保持できる、拡張性と可用性に優れたデータレイクの中核となるオブジェクトストレージサービスです。

- **試験での出題ポイント**
  - **統合データリポジトリの構築**: 多種多様なソースからデータを一元的に取り込み、フォーマットを問わず生データを保存するストレージソリューションとして、「Amazon S3 ベースのデータレイク」が最適解となるシナリオが頻出します。
  - **機械学習ツールとの統合**: S3 に保存されたデータは、Amazon SageMaker をはじめとする機械学習や分析ツール（AWS Glue, Athena など）とシームレスに統合でき、複数のチームが横断的にデータへアクセスできる構成が問われます。
- **実務でのユースケース**
  - 機械学習プロジェクトにおいて、分析チームやデータサイエンティストがアクセスする共通のデータリポジトリ（SSOT）として設定されます。各種ログ、画像データ、DBのエクスポートから、特徴量抽出前の元データを集約する基盤として利用されます。
- **副業案件での需要度と単価感**
  - データ基盤やMLOpsの構築案件（単価: 時給4,000円〜8,000円程度）において、AWS GlueとS3を組み合わせたデータレイク設計は必須とも言えるスキルであり、最も需要が高く汎用的に使える技術の一つです。

### Amazon S3 Glacier を活用したアクセス頻度の低いデータのアーカイブ

アーカイブストレージクラスである S3 Glacier シリーズは、アクセス頻度が低い長期保存データのコスト最適化に利用されます。

- **試験での出題ポイント**
  - **要件に応じたストレージクラスの使い分け**: 普段はアクセスされないが、いざ必要になった際に「ミリ秒単位での即時取得」が求められるデータ（例：四半期に一度のコンプライアンス監査対応など）には **S3 Glacier Instant Retrieval** が適しています。一方で、7〜10年以上の長期間アクセスされないが、厳格なコンプライアンスポリシーにより保持が義務付けられているデータには **S3 Glacier Deep Archive** が最もコスト効率の高いオプションとして問われます。
  - **ライフサイクルルールの設計**: 最初は S3 Glacier Instant Retrieval に保存して必要に応じた即時アクセスに備え、さらに一定期間が経過した後に S3 Glacier Deep Archive へ移行して長期保存する、といった段階的なコスト削減のライフサイクル戦略が出題されます。
- **実務でのユースケース**
  - 機械学習プロジェクトにおいて、古くなった大量のトレーニングデータや推論ログ、過去のモデルアーティファクトなどを、将来の再学習や監査の要件を満たしつつ、ストレージコストを極限まで下げる目的で利用されます。
- **副業案件での需要度と単価感**
  - AWS環境の「コスト最適化案件」（単価: 時給3,000円〜6,000円程度）で必須となる設計ノウハウです。S3のライフサイクルポリシーを適切に設定するだけでクライアントのインフラコストを劇的に削減できるため、監査や構築支援で重宝されます。

### Amazon SageMaker Data Wrangler と層化サンプリング (Stratified Sampling)

Amazon SageMaker Data Wrangler は、機械学習用データの準備（クリーニング、変換、サンプリングなど）を視覚的に行えるサービスです。

- **試験での出題ポイント**
  - **層化サンプリングによる分布の維持**: 異常検出モデルのトレーニングにおいて、大容量データからサブセットを抽出する際、「層化サンプリング（Stratified Sampling）」を選択するシナリオが問われます。これにより、元のデータの分布（正常と異常の比率）が維持され、希少な異常データが失われるのを防ぎます。
  - **トレーニング時間の短縮と精度の両立**: データセット全体のサイズを縮小しつつも異常データの割合を保つため、モデルの精度を損なうことなくトレーニング時間を大幅に短縮できる最適なアプローチとして出題されます。
- **実務でのユースケース**
  - 金融の不正検知（クレジットカードの不正利用など）や機材の異常検知モデルなど、「クラスの不均衡（Class Imbalance）」が激しいデータセットを扱う際の前処理（ダウンサンプリング等）プロセスとして重宝されます。
- **副業案件での需要度と単価感**
  - SageMakerを用いた前処理パイプラインの構築案件（単価: 時給4,000円〜7,000円程度）で必須のデータサイエンス知識です。クラス不均衡への対処は機械学習プロジェクトの成否に直結するため、適切に実装できるとクライアントの信頼を獲得しやすい強みになります。

### SageMaker JumpStart と Amazon Comprehend の比較（センチメント分析）

SageMaker JumpStartを使えば事前学習済みモデルを使ってセンチメント分析が可能ですが、Amazon Comprehendと比べると「モデルの選択」「エンドポイントのデプロイとスケーリングの設定」「プロンプトの選択・調整」といった開発の手間（オーバーヘッド）が追加で発生します。

- **試験での出題ポイント**
  - **開発労力の最小化**: 「開発の労力を最小限に抑えたい」「運用オーバーヘッドを減らしたい」という要件でセンチメント分析を行う場合は、フルマネージドAPIである **Amazon Comprehend** が最適解（正解）になります。
  - **カスタマイズの必要性**: Comprehendでは対応できない独自のカスタマイズが必要な場合や、特定のオープンソースモデルを利用したいシナリオで **SageMaker JumpStart** が選ばれます。
- **実務でのユースケース**
  - **Amazon Comprehend**: カスタマーサポートのチャット履歴やSNSレビューから手っ取り早く「ポジティブ／ネガティブ」を判定し、BIツール（QuickSight等）で可視化したい場合などに利用されます。
  - **SageMaker JumpStart**: 業界特有の専門用語をモデルに理解させたい場合や、モデルのインフラを自社のVPC内で完全にコントロールしたい（高いセキュリティ要件）場合に使用されます。
- **副業案件での需要度と単価感**
  - ComprehendなどのAIサービス（API）組み込み案件は、手軽なため時給3,000円〜4,000円程度の一般的な需要が多いです。一方、JumpStartを活用して自社用AIエンドポイントを構築・運用する案件は、インフラとAI両方の知識が求められるため、時給5,000円〜8,000円といった高単価なMLOps案件になりやすい傾向があります。

### AWS Lambda と Amazon SageMaker の比較（大量データの前処理）

AWS Lambda はサーバー管理不要でコードを実行できるサービスですが、**大量のデータ前処理には適していません**。理由は、Lambda には「最大メモリサイズ（10GB）」「最大実行時間（15分）」「デプロイパッケージサイズ（250MB）」といった厳しい制限（クォータ）があるためです。一方、Amazon SageMaker は機械学習のビルド・学習・デプロイに特化したフルマネージドサービスであり、多数の組み込みアルゴリズムと大規模なデータ処理能力を備えています。

- **試験での出題ポイント**
  - **Lambda の制約**: 「大容量のデータを1つのプロセスで変換する」「数十分〜数時間かかる処理」といった要件に対して AWS Lambda を選択する選択肢は **不正解**（上限エラーになるため）になります。
  - **SageMaker などの選択**: 大規模なデータ前処理には、SageMaker Processing、AWS Glue、Amazon EMR といった、分散処理が可能で時間制限の緩いサービスを選ぶのが最適解です。
- **実務でのユースケース**
  - **AWS Lambda**: 小規模なデータ（S3に画像がアップロードされた際のトリガー処理や、Kinesisからの軽量なストリーミングデータ加工など）のイベント駆動型処理に使用します。
  - **Amazon SageMaker**: 何GB〜数TBに及ぶログファイルやCSVのクレンジング、深層学習用の巨大なデータセットの変換など、リソースを大量に消費する重いバッチ処理に使用します。
- **副業案件での需要度と単価感**
  - Lambdaによる自動連携の設定などは非常に需要が高いです（単価: 時給3,000円〜5,000円程度）。しかし実務では「Lambdaで作ったらデータ量が増えてタイムアウトするようになった」というトラブルがよく発生します。そこでアーキテクチャの限界を見極め、GlueやSageMakerへの移行を適切に提案できると、高単価（時給5,000円〜8,000円以上）なエンジニアとして重宝されます。

### Amazon SageMaker Clarify における「ファセット (Facet)」とは

SageMaker Clarify は、機械学習データやモデルに潜む「バイアス（偏見）」を検出し、モデルの予測根拠を説明するための機能です。この中で**「どの項目についてバイアスを調べるか」を指定する対象を「ファセット」**と呼びます。

- **用語の整理**
  - **ファセット (Facet)**: バイアスを調べたいデータの属性（例：「性別」「年齢」）
  - **ファセット値 (Facet Value)**: ファセットの具体的な値（例：性別ファセットにおける「女性」「高齢者」など）
  - **ターゲット (Target)**: モデルが予測したい結果（例：「ローンの承認・非承認」）
- **試験での出題ポイント**
  - **バイアス検出とファセットの指定**: 「モデルが特定のグループに対して不公平な判定をしていないか評価したい」という要件では **SageMaker Clarify** が正解になります。その際、「特定のグループ」を定義するために設定するのが**ファセット**です。
  - **トレーニング前後のバイアス**: アルゴリズムの学習前（データセット自体のバイアス）と、学習後（モデルの予測のバイアス）の両方でファセットを指定してレポートを出力できる点が問われることがあります。
- **実務でのユースケース**
  - **金融や採用モデルの公平性検証**: 例えば「住宅ローンの審査AI」を作る際、AIが過去の偏ったデータの影響を受けて「特定の性別（ファセット）や年齢層のローンを不当に拒否していないか」を監査するために利用されます。
- **副業案件での需要度と単価感**
  - AIの倫理（Responsible AI）や公平性の担保は、大企業や金融・ヘルスケア系のプロジェクトで非常に厳しく求められます。単にモデルを作るだけでなく、「SageMaker Clarifyを使ってファセットごとのバイアスレポートを作成し、非エンジニア（顧客や監査担当）に公平性を証明できる」エンジニアは希少価値が高く、時給5,000円〜8,000円、あるいはそれ以上の高単価なデータサイエンティスト／MLOps案件で重宝されます。

### SageMaker Pipelines と Amazon EMR の統合（CI/CDパイプライン）

SageMaker Pipelines を使用してモデルの自動ビルド（CI/CDパイプライン）を構築する際、大規模なデータ前処理（Hadoop / MapReduce / Sparkなど）を Amazon EMR で行いたい場合のアーキテクチャ設計についての知識です。

- **試験での出題ポイント**
  - **EMR Step の使用**: SageMaker Pipelines の中で EMR クラスターを作成・実行するためには、処理ステップとして `SageMaker Processing Step` ではなく、ネイティブに統合された **`Amazon EMR Step`** を使用するのが正解です。
  - **IAMロールの割り当て**: パイプライン内で EMR クラスターを作成・呼び出す権限（IAMポリシー）は、データサイエンティストが「SageMaker Studio を操作しているロール」ではなく、「**SageMaker Pipelines 自身に割り当てられた実行ロール**」に付与されている必要があります。
- **実務でのユースケース**
  - **大規模前処理を含むMLOpsパイプライン**: 深層学習モデルのビルド前に、数テラバイトのログデータを EMR (Spark / MapReduce 等) で分散処理（クレンジングや特徴量エンジニアリング）し、そのまま自動的に SageMaker での学習ステップ、評価ステップ、Model Registry 登録へと繋げる「完全自動化されたワークフロー」の構築に使用されます。
- **副業案件での需要度と単価感**
  - 単発のAIモデル開発ではなく、「SageMaker Pipelinesと既存のデータ基盤（EMR等）の統合を完全自動化する」ことのできるアーキテクトは極めて希少です。AWSのビッグデータサービス群とMLサービス群を組み合わせたMLOps構築案件は、時給6,000円〜10,000円クラスの非常に高単価なプロジェクトになりやすいです。特に、環境ごとのIAMロールの適切な切り分け（Studio用、パイプライン用、各ジョブ用）などの知見は実務で常に重宝されます。

### SageMaker Processing とオンプレミスDB（IP制限あり）のセキュアな接続

社内のオンプレミス環境や他社クラウドにあるデータベースから、独自のファイアウォール（IPアドレスの許可リスト）による制限を突破して SageMaker Processing でデータを取得するためのネットワーク設計です。

- **試験での出題ポイント**
  - **NAT Gateway + Elastic IP の活用**: SageMaker の処理ステップを**プライベートサブネット (`VpcConfig` を利用)** で起動し、そのアウトバウンドトラフィックを**パブリックサブネットの NAT Gateway** にルーティングするのが正解シナリオです。NAT Gateway に割り当てられた固定のパブリックIP（Elastic IP）を社内ファイアウォールの許可リストに追加することでアクセス可能になります。
  - **不正解選択肢の理由**: パブリックサブネットで処理ステップを直接起動しても、起動のたびに割り当てられるパブリックIPが変わってしまうため許可リストに事前追加できません。また、AWS PrivateLink は外部からAWSへのアクセスで使われるサービスであり、VPC内から外部データベースへの通信要件には合致しません。
- **実務でのユースケース**
  - **ハイブリッド環境におけるデータ連携**: 大企業の分析プロジェクトでよくある「AWS上で機械学習を行いたいが、顧客のメインデータはまだ自社のオンプレミスDBにある」というケースにおいて、最も標準的でセキュアなネットワーク設計パターンとして利用されます。
- **副業案件での需要度と単価感**
  - 「VPC・サブネット・NAT Gateway・Elastic IP」といった本格的なネットワーク知識を含んだ機械学習基盤の構築は、インフラエンジニアの領域もカバーするため高単価（時給5,000円〜7,000円程度）なアーキテクト案件になります。「なぜパブリックサブネットに直接置いてはいけないのか」を顧客に説明し、セキュアな構成を提案・実装できると非常に信頼されます。

### SageMaker 各種ツールの役割と使い分け（MLOps関連）

機械学習（MLOps）の自動化要件においてよく問われる、名前の似たSageMaker関連ツールの正確な役割と、できること・できないことの違いです。

- **SageMaker Projects**
  - **役割**: MLOpsソリューション全体の包括的な管理システム・テンプレート
  - **特徴**: 「依存関係管理」「コードリポジトリ管理」「ビルドの再現性担保」「成果物の共有機能」など、**CI/CDシステム全般の標準化機能**を提供します。（StudioやPipelinesなどをラップして包括的な環境を作る機能です）
- **SageMaker Studio**
  - **役割**: データサイエンティスト向けの統合開発環境（IDE）
  - **特徴**: コードの執筆、モデルのトレーニング・デバッグ・デプロイなどに使用される作業画面です。**CI/CDパイプラインの標準化やソースコードのリポジトリ管理などを担うツールではありません**。
- **SageMaker Pipelines**
  - **役割**: 機械学習ワークフローのオーケストレーター（自動化ツール）
  - **特徴**: 「前処理 → 学習 → 評価 → 登録」といった一連のMLタスクをシーケンシャル（または並列）に**自動実行（オーケストレーション）**します。ただし、リポジトリ管理やビルド再現性の担保、依存関係の管理といった機能は単体では持っていません。
- **推論パイプライン (Inference Pipeline)**
  - **役割**: 推論における複数コンテナの連結（直列実行）
  - **特徴**: 「データ前処理コンテナ → 推論モデルコンテナ」のように、2〜15個のコンテナを組み合わせて1つのモデル（エンドポイント）として動かす仕組みです。**CI/CDの自動化やコード管理とは全く関係ありません**。

- **試験での出題ポイント**
  - **機能のひっかけ問題**: 「依存関係管理・コードリポジトリ管理・ビルド環境の標準化」が必要な要件において、SageMaker Studio や SageMaker Pipelines、推論パイプラインを選択するのは間違い（不正解）です。この要件をすべて満たせるのは **SageMaker Projects** のみとなります。
- **実務でのユースケースと副業単価**
  - CI/CDやリポジトリの準備（Projects基盤）から、実際の学習経路の自動化（Pipelines）までを自力で設計・構築できるMLOpsスペシャリストは市場で圧倒的に不足しています。時給6,000円〜10,000円クラスの高度な基盤構築コンサルティング案件に直結する知識です。

### SageMaker Experiments の役割と制約

SageMaker Experiments は、機械学習モデルのトレーニング履歴（各試行でのハイパーパラメータ設定や、モデルの精度など）を記録・分析・比較するための**実験管理（Experiment Tracking）ツール**です。

- **試験での出題ポイント**
  - **できないこと（ひっかけ問題）**: SageMaker Experiments は学習時の「指標スコア」を比較するツールであり、**「Jupyter Notebookのソースコードの変更履歴（バージョン管理）」や「ノートブックのメンバー間での直接共有・共同作業」をサポートする機能は持っていません。** ノートブックのバージョン管理には Git連携 が必要になります。
- **実務でのユースケース**
  - データサイエンティストが「パラメータを少しずつ変えて100回学習を回した結果、どの設定のモデルが一番精度が良かったか」を一覧表やグラフで比較・可視化し、最適なモデルを選定するために利用されます。
- **副業案件での需要度と単価感**
  - 実験管理の仕組み化は、属人的になりがちなデータサイエンス業務において必須となるため需要が高いです（時給4,000円〜7,000円程度）。すでにOSSの「MLflow」や「Weights & Biases」で実験管理を行っているチームの基盤を、SageMakerエコシステムに移行（統合）するようなプロジェクトで重宝されます。

### SageMaker Model Monitor と SageMaker Clarify の監視（ドリフト・バイアス）

運用フェーズに入った機械学習モデルの品質低下（ドリフト）や偏り（バイアス）を自動的に監視するツールの役割です。

- **SageMaker Model Monitor**
  - エンドポイントにデプロイされたモデルの品質を、継続的またはスケジュールベースで監視します。
  - **試験での出題ポイント**: スケジュール設定時に `enable_cloudwatch_metrics` を `True` に設定することで、データ品質やモデル精度の低下（データドリフトなど）を CloudWatch アラームとして発行・通知できる連携手法が出題されます。
- **SageMaker Clarify**
  - 機械学習データやモデルに潜む「事前学習バイアス」や「事後学習バイアス」を検出し、モデルの公平性を検証ならびに説明（Explainable AI）するための専用ツールです。
  - **試験での出題ポイント**: 単なる予測精度の低下ではなく、「特徴量の重要度が時間とともに変化する（特徴の帰属ドリフト）」という状況を監視するためには、Clarify の機能である **SHAP ベースライン** を使用して特徴量の影響（Feature Attribution）を測定するのが正解になります。
- **実務でのユースケースと副業単価**
  - 「AIモデルを本番環境へデプロイして終わり」ではなく、数ヶ月後に発生する「入力データの傾向変化（ドリフト）」や「推論の偏り」を自動検知して再学習を促す仕組み（MLOpsの要）を作るために利用されます。AIやMLシステムのライフサイクル全体を適切に監視・保守運用できるエンジニアは需要が高く（時給5,000円〜8,000円程度）、長期稼働のインフラ・運用案件も獲得しやすくなります。

### SageMaker Inference Recommender による推論コストの最適化

SageMaker にデプロイするモデルに対して自動的に負荷テストを行い、要件を満たしつつ最もコスト効率の良い（無駄なリソース使用を削れる）エンドポイント用EC2インスタンスタイプを推奨してくれる機能です。

- **試験での出題ポイント**
  - **コスト最適化と性能の保証**: 「エンドポイントのパフォーマンス（レイテンシーやスループット）を損なうことなく、利用率の低いEC2インスタンスの無駄を省き、**コストが最も低いインスタンスタイプを選択したい**」という要件において、**SageMaker Inference Recommender** が正解になります。
  - **負荷テストの自動化**: エンジニアが手作業で複数のインスタンスを立ち上げて負荷テストをするのではなく、「推奨ジョブ」を1つ作成するだけで自動的に様々なインスタンスを評価してくれる自動化ツールとしての性質が問われます。
- **実務でのユースケース**
  - **本番デプロイ前のサイジング**: 新しいモデルを24時間稼働の本番環境へデプロイする直前に実行し、「どのインスタンスタイプが目標レスポンス速度を保ちつつ一番安く済むか」というインフラ選定の明確な根拠やレポートを作成するために利用します。
- **副業案件での需要度と単価感**
  - お客様のAWS料金を直接下げる「コスト最適化・インフラ見直し案件」（単価: 時給3,000円〜6,000円）で非常に強力な武器になります。推論用のEC2エンドポイント構成は適当に大きなサイズが選ばれがちですが、このツールを使って正確なサイジングを提案し、毎月の無駄なインフラ費用を削ることができれば顧客の信頼を一気に獲得できます。

### SageMaker 推論エンドポイントの Auto Scaling（オートスケーリング）戦略

トラフィックの増減に合わせてエンドポイントのインスタンス数を自動調整（スケーリング）する際、**どのメトリクス（指標）を基準にするべきか**というベストプラクティスです。

- **試験での出題ポイント**
  - **「呼び出し数」基準 vs 「レイテンシー」基準**: ボトルネックによる遅延（レイテンシー）を防ぎたい場合でも、「レイテンシー（応答時間）」自体をスケーリングの基準とするより、**「インスタンスごとの呼び出し数（InvocationsPerInstance）」などの呼び出しメトリクスを基準とする**方が、より安定したパフォーマンスを提供できるため正解シナリオとなります。
- **実務でのユースケース**
  - **スパイク（突発的なアクセス増）への対応**: スマホアプリのバックエンドで動く推論APIなどで、夜間やイベント時にアクセスが急増するケースにおいて、「レイテンシーが悪化し始めてから後手でサーバーを増やす」のではなく、「リクエスト数（呼び出し数）が増えた時点でトラフィックに比例してサーバーを増やす」という安定したインフラ設計のために利用されます。
- **副業案件での需要度と単価感**
  - 機械学習モデルのWeb API化や本番運用案件（単価: 時給4,000円〜7,000円程度）で必須となるインフラ知識です。不適切なスケーリング設定によるAPIのダウンや劣化を防ぐため、「なぜレイテンシーではなく呼び出し数をトリガーにするのか」を顧客に論理的に説明し、堅牢なシステムを構築できると高く評価されます。

### SageMaker ModelQualityMonitor の役割と限界（特徴量の貢献度との違い）

`ModelQualityMonitor` クラスは、モデルの予測精度（真陽性率、F1スコア、RMSEなど）が事前定義したベースラインから低下していないか（モデル品質ドリフト）を監視するためのツールです。

- **試験での出題ポイント**
  - **精度の低下 vs 特徴量の貢献度の変化（ひっかけ問題）**: 試験問題において要件が「特徴量の重要度（貢献度）の時間的な変化を検出したい」である場合、`ModelQualityMonitor` を選択するのは**不正解**になります。
  - **正しい使い分け**: 
    - モデル全体の**予測精度・パフォーマンス**が落ちていないかを監視する → **`ModelQualityMonitor`** の役割
    - 予測結果に対して**「どの特徴量が一番影響を与えているか」の変化（特徴の帰属ドリフト）**を監視する → **SageMaker Clarify（SHAP ベースライン）** の役割
- **実務でのユースケース**
  - デプロイ後のモデルが「実際の現場から後日得られた正解データ（グラウンドトゥルース）」と比較して、どの程度正しく予測できているかを継続的にレポーティングし、再学習のトリガーを引くために利用されます。
- **副業案件での需要度と単価感**
  - MLOpsの運用監視パイプライン構築案件において、Clarifyと並んで導入されるツールです。ドリフトには種類があり、「データ自体の変化（データドリフト）」「モデル精度の低下（モデル品質ドリフト）」「各項目の影響度の変化（概念ドリフト/特徴量ドリフト）」を適切に使い分けてダッシュボード化できるエンジニアは、高度な保守案件（時給4,000円〜7,000円程度）で重宝されます。

### Amazon Textract の役割（PII・個人情報保護に関する制約）

Amazon Textract は、スキャンしたドキュメント（PDFファイルや画像ファイル）内の入力されたテキスト、手書きの文字、表、フォームのデータを高い精度で自動抽出（OCR）するサービスです。

- **試験での出題ポイント**
  - **個人情報（PII）の検出は得意ではない**: Textract はテキストの「抽出」に特化しており、**「個人情報（PII: 氏名、住所、電話番号など）を検出して編集（黒塗り・マスキング）したい」という要件で Textract 単体を選択するのは不正解（ひっかけ）**になります。高度な個人情報保護や墨塗り機能を利用するには、**Amazon Comprehend**（または医療情報なら Amazon Comprehend Medical）と組み合わせて使用する必要があります。
- **実務でのユースケース**
  - **請求書や帳票の入力を自動化（DX化）**: 経理部門などで、大量の紙の請求書や領収書をスキャンし、人間が手作業で打ち込む代わりに Textract を使ってキーとバリュー（「合計金額」と「10,000円」など）を抽出し、自社システム（ERP等）に自動入力する業務効率化によく利用されます。
- **副業案件での需要度と単価感**
  - 「紙業務のDX（デジタルトランスフォーメーション）」案件において Textract は非常に需要が高く、導入ハードルも低いため時給3,000円〜5,000円クラスのシステム内製化支援案件でよく使われます。さらに、Textractで読み取った後にComprehendを使って重要機密事項だけを検知・マスキングする複合的なパイプラインを組めると、セキュリティ要件の厳しい高単価案件（金融・医療界隈等）にアプローチしやすくなります。

### AIの解釈可能性（XAI）とバイアス検出の指標（SHAP, PDP, DPL）

SageMaker Clarify などで利用される、モデルの「判断根拠（解釈可能性）」を測定するためのツールと、「不公平性（バイアス）」を検出する指標の定義と違いです。

- **用語の整理と試験での出題ポイント**
  - **Shapley 値 (SHAP)**: 「どの特徴量が予測結果に対して一番影響を与えたか（**特徴量の貢献度**）」を測る指標。
  - **PDP (部分依存プロット: Partial Dependence Plots)**: 「興味のある1つの特徴量の値が変化した時、**予測されるターゲット結果がどう変化するか**」の全体的な傾向・依存関係を示す指標。
  - **DPL (ラベルにおける割合の差: Difference in Proportion of Labels)**: 不均衡なデータセットに潜む**「事前トレーニングのバイアス（学習を行う前にあるデータ自体の偏見）」を検出**するためのメトリクス。
  - **出題パターン（ひっかけ）**: 「学習前のバイアスを検出したい」という要件なら **DPL** が正解に。「特徴量の貢献度」を見たいなら **Shapley (SHAP)** が正解に、「ある変数が結果に与える変化の振る舞い」を見たいなら **PDP** が正解になります。キーワードでの使い分けが必須です。
- **実務でのユースケース**
  - **「責任あるAI」と監査対応**: 金融ローンの審査モデルや採用AIなどにおいて、「なぜローンを否決したのか」という顧客への説明責任（SHAPを利用）を果たし、かつ「過去の人間の偏見（性別や年齢による差別）が学習データに混ざっていないか（DPLを利用）」を証明し、公平な運用を行うために利用されます。
- **副業案件での需要度と単価感**
  - ブラックボックスになりがちな機械学習モデルにおいて、「Explainable AI (XAI)」の知見を持つエンジニアは非常に希少です。高精度なモデルを作るだけでなく、「ステークホルダーに論理的かつ定量的にモデルの公平性を説明できる」データサイエンティストは、時給6,000円〜10,000円クラスの高い単価でコンサルティング案件を獲得しやすくなります。

### SageMaker Pipelines とバッチ変換の定期実行（コスト最適化）

推論リクエストがリアルタイム（常時ストリーミング）ではなく、「定期的（例：週に1回）」に行われるようなユースケースにおいて、推論コストを最小化するためのアーキテクチャ設計です。

- **試験での出題ポイント**
  - **リアルタイム推論 vs バッチ変換**: 「週に一度など定期的に推論を行いたい」という要件に対し、24時間インスタンスが起動したままになる「リアルタイム推論エンドポイント」を選択するのは**間違ったアプローチ（コストの無駄）**として出題されます。対象データを一括で処理し、処理が終わるとすぐにインスタンスを削除して課金を止める**「バッチ変換 (Batch Transform)」**を使用するのが最もコスト効率の良い正解シナリオになります。
  - **EventBridge によるスケジューリング**: パイプラインを「毎週の始め」のように定期実行するには、EventBridge の**スケジュールルール（cron式など）**を使用します。この際「カスタムイベントパターン」を作る必要はありませんが、EventBridge が SageMaker Pipeline を実行できるようにするための**適切なIAMロール（実行許可）の作成・付与**が必須になる点が知識として問われます。
- **実務でのユースケース**
  - **大規模な夜間・週末バッチ処理**: 実社会で非常によくある「毎週月曜日の朝に、全店舗の向こう1週間の需要予測を計算してデータベースに格納する」「毎日深夜に、全ユーザー向けのレコメンドリストを一括生成する」といった、リアルタイム性を求められない大規模な機械学習タスクで標準的に利用される運用パターンです。
- **副業案件での需要度と単価感**
  - バッチ推論パイプラインの構築案件は非常に引き合いが多く、基礎的かつ汎用的なスキルです（単価: 時給3,000円〜6,000円程度）。「リアルタイム推論の維持費がかさむ」という顧客の悩みを、バッチ変換とEventBridgeによる自動定期実行に置き換えることでインフラコストを劇的に最適化できるため、顧客満足度の高い提案が可能になります。

### SageMaker エンドポイントの Data Capture（データキャプチャ）機能

本番稼働しているモデルの推論エンドポイントに対して、ユーザーからリアルタイムに送られてきた実データ（入力）と、その予測結果（出力）を自動的に Amazon S3 に記録する構成です。

- **試験での出題ポイント**
  - **本番トラフィックへの影響ゼロ（非同期実行）**: 試験では「本番環境の応答速度（レイテンシー）に悪影響を与えずにデータを記録したい」というよくある要件が出題されます。Data Capture 機能は推論プロセスに対して**非同期**でデータをS3に保存するため、本番トラフィックのパフォーマンスに一切影響を与えないのが重要な正解ポイントです。
  - **Model Monitor と再学習への活用**: キャプチャされた生データは、そのまま SageMaker Model Monitor に読み込まれて「データドリフト」などの監視に使われます。さらに、この最新のデータを蓄積しておくことで、将来的に**モデルの再学習（リトレーニング）用データ**としても二次利用できる点が問われます。
- **実務でのユースケース**
  - **継続的なモデル改善（Continuous Training）**: AIをリリースした直後は綺麗なデータで学習されていますが、時間が経つとユーザーの入力傾向が変わります。そのため、Data Capture で本番のリアルな入力ログを蓄積し続け、それを使ってモデルをアップデート（再学習）するループを作ることが MLOps において非常に重要になります。
- **副業案件での需要度と単価感**
  - 単なる推論環境の構築ではなく、「データキャプチャによる蓄積 → Model Monitor による監視 → パイプラインでの自動再学習」という一連の MLOps ループ基盤を設計できるエンジニアは、時給6,000円〜8,000円以上の高単価なインフラ構築コンサルティング案件において圧倒的な需要と高評価を得られます。

### SageMaker 非同期エンドポイント（コスト効率と長時間処理）

SageMaker における推論ホスティングの選択肢の1つで、すぐに結果を返す必要がない、処理の重い推論タスクに特化したエンドポイントアーキテクチャです。

- **試験での出題ポイント**
  - **最大60分の処理とコスト削減**: 「1つの推論リクエストに最大60分かかる」ような重い要件に対し、リアルタイムエンドポイントを選ぶとタイムアウト（60秒制限）するため不正解になります。また、非同期エンドポイントは**インスタンス数を自動的にゼロまでスケール（ゼロスケーリング）**できるため、リクエストがない時間帯の**アイドルコストがかからない**、最もコスト効率に優れたソリューションとして出題されます。
  - **VPC内への配置**: コスト最適化だけでなく、金融等の機密要件に合わせて VPC 内へ接続・デプロイすることも可能である点が、セキュリティとコストの両立シナリオとして問われます。
- **実務でのユースケース**
  - **重いコンピュータビジョン・NLP処理タスク**: 「ユーザーがアップロードした高画質の長尺動画から数十分かけて物体を検知する」「大容量のPDF資料から長文要約を生成する」といった処理において、リクエストだけS3経由で投げておき、結果の完了を待つ（終わったらSNS等で通知を受け取る）バックグラウンド処理用のAPIとして最も利用されます。
- **副業案件での需要度と単価感**
  - Web・アプリのバックエンド設計案件において、「AIサーバーを立てっぱなしにするとAWS代が高い」と悩むクライアントに対して、非同期エンドポイントによる**「ゼロスケーリング（使った分だけ課金）」構成**を提案できると非常に強い説得力を持ちます（単価: 時給4,000円〜6,000円程度）。タイムアウトの落とし穴を防ぎつつコストを最適化できるアーキテクトは厚く信頼されます。
