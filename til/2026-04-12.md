## 学習内容

### AWS Glue
- **概要**: サーバーレスのデータ統合サービスで、S3など複数ソースからのデータをApache Sparkエンジン等を用いて統合・処理する。
- **試験ポイント**: ETL（抽出・変換・ロード）の自動化。サーバーレスでインフラ管理が不要なため、「最小限の運用労力」で要件を満たすといったシナリオで正解になりやすい。
- **実務での使い方**: データレイク（S3等）と連携し、機械学習モデル開発前の前処理や特徴量エンジニアリング、定期的なバッチ処理（ETLパイプライン）の構築に広く利用される。
- **副業での需要**: ★★★★☆ ETLパイプラインの構築・自動化やデータ基盤整備の案件で非常に需要が高い。分析用データマート作成などの定常業務自動化タスクとして活躍する。

### AWS DMS & SageMaker 学習用データソース
- **概要**: AWS DMSはデータベースの移行や複製を行うマネージドサービス。SageMakerの学習用データソースとしてはS3、EFS、FSx for Lustreのみがサポートされており、RDSは直接指定できない。
- **試験ポイント**: SageMakerのデフォルトの入力モードは「S3ファイルモード」である点、またRDS等のデータベースから直接学習用データを取得することはできず、DMS等でS3などに抽出する必要がある点が問われやすい。
- **実務での使い方**: 運用中のRDB（MySQL/PostgreSQL等）にあるデータをDMSを使ってS3にエクスポート（データレイク化）し、そこからSageMakerの学習ジョブに連携させるデータパイプラインを構築するのが一般的な構成。
- **副業での需要**: ★★★☆☆ 既存データベースから機械学習基盤や分析基盤へのデータ抽出エンジニアリング案件として需要がある。

### Amazon EMR & Apache Spark
- **概要**: EMRは、Sparkなどの分散処理フレームワークを使用して大規模なデータ処理や機械学習を実行できるマネージドクラスターサービス。
- **試験ポイント**: EMR上のSparkを利用することで、大量のデータを並行処理し、SageMakerでの学習に最適化された軽量なRecordIO-Protobuf形式に変換できる点が有効な選択肢として出題される。
- **実務での使い方**: テラバイト〜ペタバイト級の大規模データセットの前処理や変換において、サーバーレスのGlueでは柔軟性やコスト効率が合わず、細かいSparkチューニングが必要な場面でEMRを採用する。
- **副業での需要**: ★★★☆☆ 大規模データ基盤案件での需要はあるが、小中規模であればインフラ管理不要なAWS Glueが選ばれることが多いため、ピンポイントな専門タスクになりやすい。

### SageMaker 特徴量グループ (Feature Store)
- **概要**: 機械学習の特徴量を一元的に保存、共有、管理するためのフルマネージド型リポジトリ機能。
- **試験ポイント**: `PutRecord` APIを利用することで、特徴量グループの既存レコードを上書き更新できる点。最新の特徴量のみを維持し、不要になった履歴レコードを上書きするシナリオで正解となる。
- **実務での使い方**: オンライン推論システムでリアルタイムなユーザー属性データなどが必要な場合、バックエンド処理で最新の状態に更新（PutRecord）し、推論エンドポイントから低レイテンシーで参照するアーキテクチャをとる。
- **副業での需要**: ★★☆☆☆ 特徴量ストアまで導入する現場は成熟したML組織に限られるため、案件数自体は少なめだが、MLOps上級者向けの要件として高単価を狙える。

### Amazon Bedrock Knowledge Bases & RAG
- **概要**: 自社データを取り込み、関連性の高い情報に基づいた生成AIの回答を生み出すRAG（検索拡張生成）アプリを構築できる機能。
- **試験ポイント**: Knowledge Basesの初期データソースとしては「S3バケット」を指定する必要がある点。最初からOpenSearch Serviceなどをデータソースとして直接アップロードさせる選択肢は「誤り（ひっかけ）」となる。
- **実務での使い方**: 社内マニュアルやFAQなどのドキュメントをS3に配置するだけで、自動でチャンク化・ベクトル化・OpenSearch Serverless等へのインデックス化が行われるため、素早くセキュアな自社特化AIチャットボットを構築できる。
- **副業での需要**: ★★★★★ RAGを用いた社内文書QAシステム構築は現在の生成AI副業案件の定番であり、BedrockとS3を活用したサーバーレス構成の提案は非常に需要が高く高単価案件を獲得しやすい。

### SageMaker Pipelines のスケジュール実行
- **概要**: Amazon EventBridge を使用して、事前に定義されたスケジュールで SageMaker Pipeline を自動的に呼び出す機能。
- **試験ポイント**: EventBridge のアクションによって直接「SageMaker Pipeline」を呼び出すことはできるが、「SageMaker Data Wranglerのフロー」を直接スケジュール実行・呼び出す方法ではない点に注意。
- **実務での使い方**: 新たなデータがS3に到着したタイミングや日時（週次・日次など）をトリガーにして、データ前処理〜モデル学習の一連のパイプラインを自動再実行する MLOps の基本構成として利用する。
- **副業での需要**: ★★★★☆ MLOpsパイプラインの自動化はAI・ML案件の後半フェーズで必ず話題に上がり、EventBridgeと連携した自動学習ループ構築は需要も単価も高い。

### SageMaker Feature Store (オンラインストアとオフラインストア)
- **概要**: 手間のかかる前処理済みの特徴量を一元管理・再利用するためのサービス。低レイテンシーな「オンラインストア」と、バッチ学習向けの「オフラインストア」を使い分ける。
- **試験ポイント**: 用途に応じたストアの使い分けが定石。例えば「毎晩のバッチ学習（トレーニング）」にはオフラインストアを関連付け、「ストリーミングモードで取り込むデータ」のリアルタイム推論にはオンラインストアを使用する、といった構成が問われる。
- **実務での使い方**: ユーザー行動やストリーミングデータ等を取り込み、リアルタイムでのレコメンドなどミリ秒単位の予測が必要な際に推論エンドポイントからオンラインストアを参照し、過去の履歴データを使った定期的なモデル再学習にはS3バックエンドのオフラインストアを用いるシステム構成をとる。
- **副業での需要**: ★★☆☆☆ 本格的な特徴量ストア（オンライン/オフライン）の統合まで行うのは成熟したデータ組織や大規模なリアルタイムプロダクトに限定されるため、案件数は少ないがピンポイントでの募集は非常に高単価。

### SageMaker 組み込みアルゴリズム (Built-in Algorithms)
- **概要**: AWSがあらかじめ提供・管理している、一般的な機械学習タスク向けの最適化されたアルゴリズム実装。
- **試験ポイント**: 組み込みアルゴリズムを使用する場合でも「コードを使用してトレーニングデータ、環境、ステップを定義する必要がある」点が問われやすい。AutoPilot等の完全な「ノーコードソリューション」ではない点に注意。
- **実務での使い方**: 自社でスクラッチからアルゴリズムを実装（カスタムコンテナ等）するにはコストが見合わない場合や、標準的なタスク（XGBoostでの分類、K-Means等）をSageMaker上でいち早くPoC検証するために利用する。
- **副業での需要**: ★★★☆☆ フルスクラッチのカスタムモデル開発よりも、AWSが独自に最適化・提供している組み込みアルゴリズムを利用することで、素早く確実にシステムを納品でき、コストパフォーマンスが良い。

### SageMaker Model Cards
- **概要**: モデルの目的、アーキテクチャ、トレーニングの詳細、評価指標などのビジネス上のメタデータを1か所で文書化・管理できる機能。
- **試験ポイント**: コンプライアンス要件やステークホルダーとの共有目的で、「モデルの詳細やビジネス上の背景情報を1か所で文書化し、エクスポートして共有する」シナリオでModel Cardsが正解となる。
- **実務での使い方**: モデルのバージョン管理（Model Registry）と合わせてModel Cardsを作成し、データサイエンティスト間やビジネス側のステークホルダーとの間でモデルの仕様（使用制限、バイアスなど）を共有するためのガバナンスツールとして活用する。
- **副業での需要**: ★☆☆☆☆ 厳格なモデルガバナンスと説明責任を求める金融系・医療系の機械学習プロジェクトなどのエンタープライズ領域で稀に要件に含まれる。案件自体は少ないが高単価になりやすい。

### SageMaker リアルタイム推論と自動スケーリング (Auto Scaling)
- **概要**: SageMakerでホストされたリアルタイムエンドポイントのバックエンドインスタンス数を、ワークロードの変動に応じて動的に増減させる設定。
- **試験ポイント**: トラフィックの「予想される急増」に対してエンドポイントの可用性を高めつつ対応する最適なソリューションとして、自動スケーリングの構成が頻出する。
- **実務での使い方**: Webサービス等で日中と夜間、あるいはイベント時にアクセス数が変動する場合に適用し、リソース不足によるエラーを防ぎつつアイドル時のコスト最適化を図る。
- **副業での需要**: ★★★☆☆ 本番環境のモデル運用において必須のインフラ知見であり、運用コスト削減の提案として自動スケーリング設定を巻き取るタスクの需要は安定して存在する。

### SageMaker サーバーレス推論 (Serverless Inference)
- **概要**: サーバーのプロビジョニングや管理作業が不要で、リクエスト数に応じて自動的にスケールする推論オプション。
- **試験ポイント**: 「コールドスタートが許容される場合」の選択肢として有効である一方、処理ペイロードが大きかったり、処理時間が長くて「タイムアウト」の懸念があるシナリオでは不適切である（バッチや非同期が必要）という判断が問われる。
- **実務での使い方**: トラフィックが散発的で予測不可能な社内ツールや、レイテンシ要求が厳しくない軽量モデルのPoC環境において、インスタンスの常時立ち上げコストを回避するために採用する。
- **副業での需要**: ★★★☆☆ 初期フェーズのスタートアップやMVP開発案件において、運用コストを劇的に下げる提案としてサーバーレス推論の導入は非常に喜ばれるためスキルとしての価値が高い。

### SageMaker 非同期エンドポイント (Asynchronous Inference)
- **概要**: リクエストをキューに入れ、ほぼリアルタイムから数十分といった時間のかかる推論を非同期で実行する推論オプション。
- **試験ポイント**: 非同期推論は「最大1時間（※仕様変更などに注意）」程度の処理時間をサポートするため、例えば「90分の処理」のような長大な要件には該当せず、その場合はバッチ変換（Batch Transform）を利用する必要があるといった制約が問われる。
- **実務での使い方**: ペイロードが大きい画像・動画処理や、時間のかかる複雑な最適化推論などを捌く際に利用し、クライアント側はS3への出力やAmazon SNSの完了通知を受け取る非同期のWebシステムアーキテクチャをとる。
- **副業での需要**: ★★☆☆☆ 重いモデル処理をWebサービスに組み込む案件でアーキテクチャ設計から任される場合に必要となるが、バックエンド実装の複雑度が上がるため中〜上級者向けのスキル。

### SageMaker マルチコンテナエンドポイント
- **概要**: 1つのエンドポイント（インスタンス）の中に、異なるフレームワークやモデルを用いた複数のコンテナをホストし、パイプライン的に実行したりリソースを共有して効率化する機能。
- **試験ポイント**: 専有ホストよりもリソース使用率が向上し「費用対効果が高くなる」メリットがある。しかし、「全て同じフレームワーク（同じ環境）」でトレーニングされた複数モデルであればマルチコンテナではなく「マルチモデルエンドポイント」が適しており、無理にコンテナを分ける必要はないといった要件の切り分けが問われる。
- **実務での使い方**: PyTorchとTensorFlowなど異なるフレームワークのモデルをつなぐ推論パイプライン（直列推論）を、単一エンドポイント内でネットワークオーバーヘッドなく低レイテンシに完結させたい場合などに利用する。
- **副業での需要**: ★☆☆☆☆ 運用のコスト削減として有効だが、モデル構成・デプロイが複雑化するため、高度に最適化された推論アーキテクチャが求められる専門的な一部の案件に限られる。

### ターゲット追跡スケーリングポリシー (Target Tracking Scaling Policy)
- **概要**: 指定したメトリクス（例: CPU使用率やリクエスト数）の目標値を維持するように、必要なインスタンス数を自動的に増減させるスケーリング方式。
- **試験ポイント**: 「トラフィックのスパイク対応」において、目標値からのズレに応じて動的に機能するため、トラフィックの急増時でもスケーリングが停止することなく適切に対応できる点がソリューションとして選ばれる。
- **実務での使い方**: 推論エンドポイントの「1インスタンスあたりの平均リクエスト数」などをターゲットメトリクスに設定し、アクセス負荷の増減に応じてリアルタイムにインスタンス数を調整する設定を行う。
- **副業での需要**: ★★★☆☆ 推論環境のインフラ運用コスト最適化・自動化の提案として、Auto Scalingのターゲット追跡ポリシーの設定は定常的な需要がある。

### Amazon Bedrock プロビジョンドスループット (Provisioned Throughput)
- **概要**: Amazon Bedrockで大規模な定常的ワークロード向けに、特定のモデルコンピュートユニットを事前に購入し、保証されたスループットを確保する機能。
- **試験ポイント**: チャットボットなどが「常に使用され、いつでも利用できる必要がある（恒常的な利用）」といったユースケースにおいて、最も「費用対効果の高い」ソリューションとしてプロビジョンドスループットが正解になる。
- **実務での使い方**: 社内全従業員が常に利用するような大規模な自社特化RAGシステムなど、推論ボリュームが巨大かつ安定して発生する環境において、オンデマンド料金によるコスト爆発を防ぎ、レイテンシを担保するために利用する。
- **副業での需要**: ★★☆☆☆ 大手企業の生成AI本格運用フェーズでのみ必要になるアーキテクチャであり、PoCや立ち上げフェーズの多い副業案件ではオンデマンド利用が主流。

### SageMaker Pipelines: TrainingStep と @step デコレーター
- **概要**: Pipelinesにおいて、モデル学習等の標準タスクを実行するのが `TrainingStep`、AWS SDKにないPythonのカスタム関数をそのままパイプラインステップ化するのが `@step` デコレーター。
- **試験ポイント**: `TrainingStep` には「独自のカスタムロジックを直接実装することはできない（エスティメーターと入力のみ受け取る）」という制約、および独自の処理関数をパイプラインに結合するには `@step` デコレーターを使用する点が引っ掛けとして問われる。
- **実務での使い方**: 標準の学習処理は `TrainingStep` で実行し、学習前後の細かなデータ変換や各種APIへの通知、独自形式でのエクスポートなどカスタムロジックを実行したい場合に `@step` を使って柔軟に組む。
- **副業での需要**: ★★★★☆ Pythonネイティブな `@step` を使った柔軟なパイプライン構築は保守性が高く、昨今のMLOps自動化案件において主流の開発手法となっている。

### AWS CodePipeline と CodeBuild (MLOps向けCI/CD)
- **概要**: CodePipelineはバージョン管理や承認プロセスを含むリリースのワークフローを管理・保存し、CodeBuildはソースのコンパイル、テストの実行、アーティファクトの作成を行う。
- **試験ポイント**: CI/CDにおける役割分担が重要。リポジトリ管理ツールとの統合や「承認メカニズム」はCodePipelineが担い、ユニットテストの実行や本番用コンテナ（アーティファクト）の「構築（Build）」はCodeBuildが担当する。
- **実務での使い方**: MLOpsの実装において、モデルトレーニングコードがGitHubやCodeCommitにプッシュされた際、CodePipelineが起動して承認フローを回し、CodeBuildでテストとコンテナイメージのビルドを実行してデプロイを自動化する。
- **副業での需要**: ★★★★☆ 既存のMLモデルの継続的インテグレーション・デリバリー（CI/CD化）案件は非常に多く、CodePipelineとCodeBuildを用いたパイプライン構築スキルはインフラ寄り案件で重宝される。

### SageMaker ローリングデプロイ (Rolling Deployment)
- **概要**: 古いバージョンのインスタンスを徐々に、ステップごと・バッチごとに新しいバージョンへと入れ替えていくデプロイ戦略。
- **試験ポイント**: ステップやバッチサイズでトラフィックを管理できるが、「トラフィックの一部分（例: 10%のみ）をカナリアとして移行した状態を維持する」使い方はできない点が制約として問われる。
- **実務での使い方**: 新旧モデルを同時にホストするための余分なリソース（2倍のインスタンスなど）を確保しなくても済むため、インフラコストを抑えつつ安全にデプロイしたい場合に採用する。
- **副業での需要**: ★★☆☆☆ コストを最優先する小規模運用では使われるが、AIモデルの安全なロールバックが求められる案件ではBlue/Green展開が求められることが多い。

### SageMaker ブルー/グリーンデプロイ (Blue/Green Deployment)
- **概要**: 新旧両方の環境（インスタンス）を並行稼働させ、トラフィックを徐々に、あるいは一気に新しい環境へ切り替える手法。
- **試験ポイント**: All At Once、Canary、Linearの3つのモードがあり、「トラフィックの一部分 (Canary) のみを新バージョンに移行して問題がないか評価し、成功後に全体を移行する」要件にはこのCanary展開が正解となる。ローリング展開との明確な違いとして出題される。
- **実務での使い方**: 新しい推論モデルが本番環境の未知のデータに対してエラーを吐かないか、応答遅延がないかなどを、全体の10%のトラフィックだけ流して安全にテスト（Canaryテスト）するために利用する。
- **副業での需要**: ★★★★☆ MLOpsパイプライン構築や本番移行案件において、モデルの安全なデプロイメント戦略の設計・実装は非常に需要が高く、単価ダウンを防ぐコアスキルになる。

### SageMaker Model Dashboard
- **概要**: アカウント内にデプロイされた機械学習モデルを一元的に可視化・探索し、パフォーマンス状況を追跡するためのポータル画面。
- **試験ポイント**: ダッシュボードはあくまで「追跡・可視化」のツールであり、「ベースラインの確立」を行うためのツールではない点に注意。ベースライン自体は推論データではなく「モデルトレーニングデータ（学習時のデータ）」からジョブを回して作成する必要がある。
- **実務での使い方**: 大規模なML組織において、本番稼働中の多数のモデルを一元管理し、データサイエンティストやMLエンジニアがどのアラートが鳴っているかを俯瞰的なダッシュボード上で素早く把握するために利用する。
- **副業での需要**: ★☆☆☆☆ マネージメント向けの一元管理ツールとしての側面が強いため、小規模なモデル数が少ない副業案件ではあえて提案する機会は少ない。

### SageMaker Clarify
- **概要**: トレーニングデータや推論データの偏り（バイアス）の検出、モデルの予測根拠（説明可能性/XAI）を可視化するためのツール。
- **試験ポイント**: 「データセットのバイアス」やモデルの「説明可能性」をチェックするというキーワードが出ればClarifyが正解。また、モデル構築時だけでなく、本番環境のデータやモデルに対する継続的なモニタリングにも利用できる。
- **実務での使い方**: 金融審査アルゴリズムなど、なぜその予測結果になったのかをビジネスオーナーや監査に説明する必要があるプロジェクトにおいて、SHAP値などを用いて特徴量への影響度をレポーティングする。
- **副業での需要**: ★★★☆☆ クレジットスコア、医療診断、採用システムなど、AIの公平性（Fairness）や説明責任が厳しく問われるエンタープライズ・公共領域の高単価案件において重宝される専門スキル。

### SageMaker Debugger
- **概要**: トレーニングジョブ自体の問題を特定し、モデルのパフォーマンス（勾配消失、過学習など）をデバッグするためのツール。
- **試験ポイント**: アラートと是正措置を行うツールだが、「クラスの不均衡（Class Imbalance）」ルールはクラス間のサンプリング不均衡しか測定できず、またDebuggerがモニタリングできるのは「トレーニングジョブのみ」であり「（入力された元の）データセット全体の設定」はモニタリングできない点が引っ掛けとして出題される。（データセットのモニタリングはClarify等を使う）
- **実務での使い方**: ディープラーニングモデル（PyTorch/TensorFlow）の学習中に、テンソルデータやシステムリソースのボトルネックを継続的にキャプチャ・分析し、無駄な学習コストを未然に防ぐために使われる。
- **副業での需要**: ★★☆☆☆ ディープラーニングモデルの精度改善や学習チューニングといった、純粋なデータサイエンティスト寄り・リサーチャー寄りの案件でスコープに含まれることが多い。

### AWS X-Ray
- **概要**: マイクロサービスやサーバレス環境で構築された分散アプリケーションのパフォーマンスを分析・デバッグするためのトレースサービス。
- **試験ポイント**: 分散アプリケーションのボトルネック特定に活用されるが、「SageMaker インスタンスのシステムリソース使用率（CPU・メモリ・GPUなど）をモニタリングする」といった目的には設計されていないため不適切である点が問われる。リソース監視にはCloudWatchを使う。
- **実務での使い方**: API Gateway → Lambda → DynamoDB のようなサーバーレスアーキテクチャでどこにレイテンシが発生しているかを視覚化するために利用する。
- **副業での需要**: ★★☆☆☆ 複雑なマイクロサービス構築案件において導入をサポートすることがあるが、MLOpsの範囲ではCloudWatchの利用が多勢。

### AWS Cost Explorer とコスト配分タグ
- **概要**: AWSリソースの使用状況とコストを視覚的に追跡・分析するツール。
- **試験ポイント**: SageMaker等のリソースにユーザ定義の「コスト配分タグ（例: Project=A）」を付与し、Billing and Cost Managementコンソールで有効化することで、Cost Explorer上で「プロジェクト単位」のコストフィルタリング・追跡が可能になるという構成が正解となる。
- **実務での使い方**: 複数部署や複数プロジェクトが混在するAWS環境において、リソースのTerraformやCloudFormationテンプレートに必ずタグを埋め込み、請求を部署ごとに可視化する（FinOps）。
- **副業での需要**: ★★★☆☆ AWS環境のコスト最適化・管理案件として需要があり、タグ付けルール整備からダッシュボード構築までのタスクとしてよく含まれる。

### S3 ゲートウェイ VPC エンドポイント (Gateway VPC Endpoint)
- **概要**: VPCからインターネット（NATゲートウェイ等）を経由せずに、同一リージョン内のS3やDynamoDBへセキュアかつ無料でアクセスできるルートを提供するエンドポイント。
- **試験ポイント**: インターネットゲートウェイ等を不要にする便利な機能だが、「ゲートウェイエンドポイントは同一リージョン内でのみサポートされる」という制約が重要。そのため、「別のリージョンにあるS3のデータを保存・取得する」要件では利用できない点が引っ掛けとして出題される。
- **実務での使い方**: SageMakerの学習ジョブや推論エンドポイントなどをプライベートサブネットに配置する際、データセット置き場となるS3へセキュアにアクセスするためのネットワーク設計として必須級の構成。
- **副業での需要**: ★★★★☆ セキュリティに厳しいエンタープライズのML基盤構築において、VPCエンドポイントを活用した閉域網設計はインフラ・MLOpsエンジニアの基本スキルとして需要が高い。

### SageMaker ノートブックインスタンスの VPC ネットワーク設定
- **概要**: SageMakerのノートブックインスタンスをVPC内に配置して、セキュアにリソースにアクセスするためのネットワーク設定。
- **試験ポイント**: パブリックインターネット経由でのVPCエンドポイントへのアクセスは許可されない仕様。ノートブックからVPCエンドポイントにアクセスできないエラーが出た場合、「サブネットやセキュリティグループのルーティング設定ミス」ではなく、そもそも「ノートブックのネットワーク設定の問題」としてとらえ、ノートブックをVPC内で実行し「内部ネットワーク接続を許可する」構成が解決策になる。
- **実務での使い方**: VDIや社内VPNからのみアクセスできるよう、SageMaker Studioやノートブックインスタンスの直接のインターネットアクセスを無効にし、すべてVPCエンドポイント経由でAWSサービスにのみアクセスするガチガチのセキュリティ要件の現場で多用される。
- **副業での需要**: ★★★☆☆ 金融機関などセキュリティレベルの高い顧客案件で、インターネットに出ない（閉域での）分析環境の提供を求められる機会が多く、単価交渉の武器となる。

### SageMaker ノートブックインスタンスと NAT ゲートウェイ
- **概要**: プライベートサブネットに配置されたノートブックインスタンスから、インターネットへのアウトバウンド通信を確保するためのネットワーク構成。
- **試験ポイント**: VPCのプライベートサブネット内でホストしているノートブックから「インターネットにアクセスできない」場合の原因として、VPC内に「NAT ゲートウェイが設定されていない」もしくは「ルーティングが正しくない」点が正解として問われる。VPC内でインターネット接続が必要な場合はNATゲートウェイの配置が必須。
- **実務での使い方**: 外部のパッケージリポジトリ（PyPIなど）からライブラリをダウンロードしたり、外部API（GitHubや外部データソースなど）にアクセスする分析環境を構築する際、セキュリティ維持のためにパブリックIPを持たせずNATゲートウェイ経由で通信させる構成をとる。
- **副業での需要**: ★★★☆☆ プライベート環境でのセキュアなノートブック運用はエンタープライズの案件で基本要件となり、VPC・NAT・エンドポイントを組み合わせたセキュアなネットワーク設計知識が問われる。

### SageMaker Studio ドメインの実行ロールと権限
- **概要**: SageMaker Studioのノートブック環境から他のAWSサービス（S3、Secrets Manager等）へアクセスする際の権限を管理するIAMロールの設定。
- **試験ポイント**: ノートブックからSecrets Managerで認証情報を取得できない等のエラーが発生した場合、「SageMakerドメインにアタッチされている実行ロール（IAMロール）」に適切なポリシーがアタッチされているかを確認するアプローチが正解になる。
- **実務での使い方**: スクリプトにDBやAPIのパスワードを直書きせず、AWS Secrets Managerに保存し、SageMaker Studioの実行ロールに `secretsmanager:GetSecretValue` などの権限を付与してコード実行時に動的に・安全に取得する構成がベストプラクティス。
- **副業での需要**: ★★★★☆ セキュリティ要件の厳しい商用案件において、ハードコードの排除とIAMロールによる最小権限の原則に基づく設計は標準ルールとなっており、必須の実装スキル。

### 💡 実務向け Tips: Jupyter ノートブックのコマンド使い分け
SageMaker の Jupyter ノートブック環境（Studio や Notebook Instances）でよく使われる、シェルコマンド（`!`）とマジックコマンド（`%` / `%%`）の主な違いと使い分けです。

| 特徴 | シェルコマンド (`!`) | マジックコマンド (`%` / `%%`) |
| :--- | :--- | :--- |
| **実行主体** | OSシェル (bash等) | IPython/Jupyter (Python) |
| **状態の保持** | 一時的（次の行には影響しない） | 永続的（ノートブック内） |
| **用途** | ファイル管理、外部ツールの利用 | ディレクトリ移動、時間計測、設定 |
| **例** | `!cd /content/sample_data` | `%cd /content/sample_data` |

※ **注意点**: `!cd` コマンドでディレクトリを移動しても、その移動はその行（またはサブシェル内）でのみ有効であり、次のセルには引き継がれません。ノートブックの作業ディレクトリを永続的に変更したい場合は `%cd` を使用する必要があります。

### SageMaker Pipelines の主要なステップ一覧
SageMaker Pipelines で利用される主なステップ（Step）の役割のまとめです。試験対策としてはもちろん、実務での複雑なMLOpsパイプライン設計時に活躍する基盤知識となります。

| カテゴリー | ステップ名 | 主な役割 |
| :--- | :--- | :--- |
| **実行系 (Core Tasks)** | `ProcessingStep` | データの加工、前処理、後処理、モデル評価などを行います。 |
| | `TrainingStep` | 推定器（Estimator）を使用してモデルの学習を実行します。 |
| | `TuningStep` | ハイパーパラメータの最適化（HPO）ジョブを実行します。 |
| | `TransformStep` | 学習済みモデルを使用してバッチ推論（一括予測）を行います。 |
| **管理系 (Model Mgmt)** | `CreateModelStep` | 推論に使用する「SageMakerモデル」を作成します。 |
| | `RegisterModel` | モデルをモデルレジストリに登録し、バージョン管理を可能にします。 |
| **論理系 (Logic/Control)** | `ConditionStep` | 条件分岐（例：精度が80%以上なら登録、未満なら終了）を制御します。 |
| | `FailStep` | 意図的にパイプラインを「失敗」として終了させます（条件分岐のElseなどで使用）。 |
| **品質・解析系 (Quality)** | `QualityCheckStep` | データの統計的な偏りや、モデルの品質が低下していないかチェックします。 |
| | `ClarifyCheckStep` | バイアス検出や説明可能性（SHAP値など）の解析レポートを生成します。 |
| **外部・拡張系 (Integration)**| `LambdaStep` | AWS Lambda関数を呼び出し、簡単なカスタム処理を実行します。 |
| | `CallbackStep` | 外部システムと連携し、完了通知を受けるまで処理を待機させます。 |
| | `EMRStep` | Amazon EMR上でApache Sparkなどのジョブを実行します。 |
| | `NotebookJobStep` | Jupyter Notebook（.ipynb）をそのままジョブとして実行します。 |
