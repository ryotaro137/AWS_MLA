## JSON と JSONata の違いについて

一番大きな違いを一言でいうと、**JSONは「データの形式（名詞）」**であり、**JSONataは「データを加工するための言語（動詞）」**であるという点です。

JSONはただの「箱」ですが、JSONataはその箱の中身を取り出したり、計算したり、形を変えたりするための「道具」です。

### JSON と JSONata の比較

| 項目 | JSON | JSONata |
| :--- | :--- | :--- |
| **役割** | **データの記述**（構造化されたテキスト） | **データの変換・抽出**（クエリと言語） |
| **性質** | 静的（値が書いてあるだけ） | 動的（計算や条件分岐ができる） |
| **記述例** | `{"price": 100, "qty": 2}` | `$sum(price * qty)` |
| **Step Functionsでの用途** | 入出力データの「形」そのもの | 入力データを「どう加工して出力するか」の指示 |
| **エラーの主な原因** | カンマ忘れ、カッコの閉じ忘れなどの構文ミス | 関数名の書き間違い、パスの指定ミス |

-----

### なぜ「出力」欄がJSONataだと嬉しいのか

以前の方式（JSONPath）では、データの場所を指定すること（抽出）は得意でしたが、データの形を大幅に変える（変換）には複数のステップが必要でした。JSONataを使うと、以下のような高度な操作が**1つの「出力」欄だけで完結**します。

  * **演算:** `{"total": price * tax}` のように計算結果をそのまま出力に含める。
  * **フィルタリング:** 配列の中から特定の条件に合うデータだけを抜き出す。
  * **マッピング:** 複雑な階層構造を、全く別のフラットな構造に書き換える。
  * **スプレッド演算子:** 前回使った `...$` のように、「元のデータを全部コピーしつつ、一部だけ書き換える」といった高度な結合ができる。

### 結論

2枚目の画像で「出力」がJSONataになっているのは、**「単なるJSONを置く場所」ではなく、「JSONataという魔法を使って、次のステップに渡すデータを自由自在に作り出す場所」**に変わったからです。そのため、ただのJSONの書き方（JSONPath）をすると「そんな言葉（構文）は知らないよ」とエラー（Invalid JSON）が出てしまいます。

## JSONしか知らない人に向けたJSONataの解説

参考URL: https://blog.mmmcorp.co.jp/2025/11/26/stepfunctions-jsonata-vs-jsonpath/

### 1. 何？（JSONataとは何か）
一言でいうと、**「JSONデータの抽出・変換・計算を、簡単に行える専用のプログラミング言語（クエリ言語）」**です。

JSONは単なる「データを入れておく箱（データの形式）」ですが、入ってきたJSONデータに対して、
* 「この部分だけ抜き出したい」
* 「この配列の数値を全部足し算したい」
* 「条件によって出力する文字を変えたい」

といった**加工**を行いたい時に使う魔法の呪文のようなものが、JSONataです。

### 2. なぜ？（なぜStep FunctionsでJSONataを使うと嬉しいのか）
理由はズバリ、**「Step Functionsのワークフローが劇的にシンプルになり、Lambda関数の開発・保守がいらなくなるから」**です。

これまでのStep Functions（JSONPathという仕組み）では、単純な「データの抜き出し」はできても、「データの足し算」や「条件分岐して文字をくっつける」といった少し複雑な加工をしようとすると、以下の2パターンの対応が必要でした。

1. **めっちゃ複雑なStep Functionsのフローを組む**（Choice Stateで分岐ループを作り、1回ずつ`States.MathAdd`で足し算するなど、無駄に長いフローになる）
2. **Lambda関数をわざわざ作って処理させる**（プログラミングの手間が増え、言語のバージョンアップなどの保守作業も発生してしまう）

**JSONataが導入されたことで、これらが「Step Functionsの1つのステップ（状態）」だけで完結するようになりました。** ほぼノーコードで高度なデータ加工ができるため、圧倒的に便利になったのです。

### 3. 具体的に（どんな風に便利になったか）
ブログ記事のハンズオンの中から、JSONataの凄さが分かる具体的な例を3つ紹介します。

#### ① 配列のフィルタリングと足し算（激減したステップ数）
もし、「注文リストの中から、ステータスが"SHIPPED（発送済み）"の金額だけの合計額を出したい」という場合。

* **昔（JSONPath）:** 「合計を0で初期化」→「1番目の値を取り出して足す」→「2番目があるかチェック」→「2番目を足す」...という**ループ処理を7〜8個のステップを繋げて**自作しなければなりませんでした。
* **今（JSONata）:** ステップ内の出力欄に `{% $sum($states.input.customer.orders[status='SHIPPED'].amount) %}` と書くだけ。**たった1行**で完結します（`$sum`という便利な足し算機能が標準で使えます）。

#### ② 条件分岐と文字列の作成（Choiceが不要に）
「もし金額がしきい値以上なら『above（超え）』、未満なら『below（下回る）』という文字を含めたメッセージを作りたい」という場合。

* **昔（JSONPath）:** `Choice`という条件分岐のステップを作り、「高い場合ルート」と「低い場合ルート」の2つのステップを作ってメッセージを作成していました。
* **今（JSONata）:** 出力欄の中で直接 `金額 >= しきい値 ? 'above' : 'below'` といったプログラムのような書き方（三項演算子）ができ、文字の結合（`&`）もできるため、**分岐ステップをわざわざ作る必要がなくなりました。**

#### ③ 集計やグループ化（便利な関数が豊富）
「商品カテゴリごとの売上合計」のようなグループ化も、JSONataなら強力な関数群が用意されています。
* `$distinct()` : 被っている名前を消して重複をなくす
* `$map()` : 特定の処理を繰り返して配列を作る

こういった関数を組み合わせることで、以前ならLambdaの出番だった複雑な集計もStep Functionsだけで処理できます。
